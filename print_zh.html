Milvus 是什么

Milvus 是一个开源的分布式向量数据库：它把图片、文本、音频这类非结构化数据 先交给 AI 模型编码成向量，再在海量（十亿级）向量里按"相似度"飞快地找出最接近的若干条。 传统数据库擅长"精确匹配"（按 ID、按关键词），Milvus 擅长的是另一种问题——"哪些东西在语义上最像它"。 一个数据库专门把这件事做到又快、又准、又能横向扩展，这就是 Milvus 存在的理由。

🔌 生活类比
把传统数据库想成按书号查书的目录：你得先知道确切的编号或书名，才能精确命中一本。 Milvus 更像一位懂内容的图书管理员：你递给他一本书说"给我找几本读起来'感觉'最像这本的"， 他不靠书名，而是凭内容的相似把最接近的几本递回来。把"感觉像不像"变成可计算的"距离远不远"，正是向量数据库的看家本领。

它到底解决什么问题

我们身边绝大多数数据都是非结构化的——文本、图片、音频、视频。计算机没法直接判断"这两段话意思像不像""这两张图长得像不像"。 现代 AI 的做法是：用一个 embedding（嵌入）模型把每条数据编码成一串浮点数，也就是向量。 关键性质是：语义相近的东西，向量在空间里也彼此靠近。于是"找相似内容"就被翻译成了一个干净的数学问题—— 在向量空间里找最近邻（nearest neighbor）。

难点在规模：当数据有上亿条、每条又是几百上千维时，挨个精确算距离慢得无法接受。 工程上改用近似最近邻（ANN, Approximate Nearest Neighbor）：用专门的索引结构，牺牲一点点精度，把检索从"逐个比对"加速成"毫秒级"。 把 ANN 检索作为一等公民查询、并围绕它把存储、更新、扩展、高可用都补齐——这就是向量数据库，Milvus 是其中的代表。

🌍 宏观理解
一句话：Milvus 是一个把"ANN 相似检索"当作一等查询的数据库。它为十亿级向量而设计， 采用存算分离、云原生（K8s）的分布式架构，既能横向扩展吞吐，又能靠实时流式写入让新数据"边写边可查"。 它用 Go + C++（少量 Rust）实现，对 CPU/GPU 做了加速。

传统（标量）数据库

查询是精确匹配：WHERE id = 42、按关键词命中。 数据是结构化的行与列，索引是 B-Tree / 哈希。问"等于谁"。

向量数据库 Milvus

查询是相似检索：给一个向量，返回距离最近的 topK。 数据是高维向量（+ 标量字段），索引是 HNSW / IVF / DiskANN 等 ANN 结构。问"最像谁"。

向量从哪来：用 embedding 把万物编码成点

向量不是凭空冒出来的，而是由 embedding（嵌入）模型算出来的。可以把它想成一台"理解机"： 喂进一段文字或一张图，它吐出一串固定长度的浮点数，把这条数据的"含义"压进这串数字里。模型在海量数据上训练过， 学会了让"意思相近"的输入产出"位置相近"的向量——"猫"和"小猫"几乎挨在一起，"猫"和"利率"则隔得很远。 于是"语义相似"这件以前说不清的事，第一次有了可计算的尺子。

1

原始数据

一段文字、一张图片、一段音频……计算机眼里它们无法直接比较"像不像"。

2

embedding 模型

BERT / CLIP / bge…

训练好的神经网络，把每条数据映射成固定维度的向量。

3

向量（如 768 维）

语义相近 ⇒ 点相近；语义无关 ⇒ 点相距很远。

4

存入 Milvus

向量 + 标量字段（id、原文、价格、标签…）一起入库，建好索引等待检索。

两条要点请记牢：①维度由模型决定（常见 384 / 768 / 1536 维），同一个集合里的向量必须同维； ②灌库与查询要用同一个模型，否则两边向量"不在同一个空间"，比距离毫无意义。一句话概括分工： "把数据变成向量"是模型的事，"把海量向量存好、索引好、搜得快"才是 Milvus 的事——本教程讲的正是后者。

ANN：用一点点精度，换巨大的速度

有了向量，最朴素的搜索是暴力检索：把查询向量和库里每一条都算一遍距离，再排序取前 K。 结果精确，但太慢——1 亿条 768 维向量，一次查询要做上百亿次乘加，扛不住高并发。ANN（近似最近邻）换了个思路： 预先把向量组织成索引结构（分桶的 IVF、可导航小世界图 HNSW、面向磁盘的 DiskANN…），查询时只看一小部分候选， 用"极小的召回损失"换来"成百上千倍的提速"。

做法怎么找速度结果
暴力检索和每条都算距离慢（O(N·D)）精确
ANN 索引只看少量候选快（毫秒级）近似（召回极高）

"近似"听起来吓人，实际里召回率常在 95%~99% 以上，而速度是暴力检索的几百倍——对绝大多数应用，这是笔极划算的买卖。 Milvus 内置多种索引，让你在速度、内存、精度之间按需权衡（第 5 课会专门展开）。

把"相似"变成"距离"：一次搜索长什么样

向量就是高维空间里的一个点。两条数据像不像，用它们向量之间的距离/相似度来衡量。Milvus 支持几种常见度量： L2（欧氏距离，越小越像）、IP（内积，越大越像）、COSINE（余弦相似度）。 一次搜索就是：给定一个查询向量，在集合里找出距离最近的 K 条（称为 topK）。

一次相似检索：查询向量 q 在 4 条库内向量里找最近的 2 条（topK=2，距离越小越像）
q0.900.100.80
v10.880.120.79→距离 0.03 ✓
v20.200.950.15→距离 1.42
v30.850.050.83→距离 0.07 ✓
v40.100.300.90→距离 0.96

选哪种度量，取决于你的向量怎么来：归一化过的 embedding 常用 COSINE 或 IP（方向越一致越相似）， 关注绝对位置时用 L2。同一个集合里，建索引与搜索必须用同一种度量，否则"远近"的定义对不上。 返回的 topK 是按距离排好序的——第 1 名最像，依次往下；业务里还常配一个"分数阈值"，把"虽然最靠前、但其实不够像"的结果挡掉。

把上面这件事用 Python SDK（pymilvus）写出来，就是"建集合 → 灌数据 → 搜"三步。注意：这里展示的是用法， 后面的课会一层层钻进 Milvus 内部，看这三步在引擎里到底发生了什么。

pymilvus 快速上手create → insert → search
from pymilvus import MilvusClient

client = MilvusClient("demo.db")               # Milvus Lite：本地就是一个文件
client.create_collection(collection_name="docs", dimension=768)

client.insert(collection_name="docs", data=rows)      # rows：768 维向量 + 文本等标量字段

hits = client.search(
    collection_name="docs",
    data=[query_vector],                          # 一个查询向量
    limit=3,                                      # topK=3：返回最相似的 3 条
    output_fields=["text"],                        # 顺便把原文带回来
)

为什么需要"数据库"，而不只是一个检索库

也许你听过 FAISS、HNSWlib 这类算法库：给它一堆固定向量，它能算 ANN。那为什么还要 Milvus？因为真实业务远不止"算一次最近邻"。 一个能上生产的系统，需要把下面这些一并解决，而这恰恰是"库"与"数据库"的分界线：

算法库（如 FAISS）

在内存里对一批相对静态的向量做 ANN。 自己负责持久化、增删改、扩容、容错、并发、过滤——这些它都不管。

数据库（Milvus）

持久化与崩溃恢复、边写边查的实时更新、标量+向量混合过滤、 横向扩展到十亿级、高可用与一致性、多租户与权限——把检索变成一项服务。

把"数据库才有、而库没有"的能力摊开看，至少包括这些：

  • 持久化与崩溃恢复：进程挂了、机器宕了，数据不丢、能恢复。
  • 实时增删改：一边持续写入/删除，一边还能查到新结果，而不是"重建一次索引才生效"。
  • 混合过滤：向量相似 + 标量条件（价格、时间、标签、分区）一起约束。
  • 横向扩展与高可用：加机器即可扩容，副本与故障转移保证不停服。
  • 多租户与权限：一套集群服务很多业务，彼此隔离、各管各的权限。

一个最能说明问题的例子：在 1 亿条商品里，要找"和这张鞋子图最像、且 价格 < 500、且 当前有货"的 10 件。 纯算法库只会"按向量找最像"，没法附带"价格、库存"这些条件；Milvus 则能把标量过滤（一个布尔表达式）与向量检索 揉在一起一次做完。这类"向量 + 条件"的混合查询，恰恰是把相似检索真正落进业务的关键一步。

换句话说：算法库解决"怎么算得快"，数据库解决"怎么在真实世界里长期、可靠、规模化地用起来"。 Milvus 内部确实也用到了向量检索的内核库（Knowhere），但在它之上补齐了数据库该有的一切。

Milvus 都用在哪：典型场景

把"按相似度找"这一种能力用起来，能解决一大批看似不同、本质相通的问题：

检索增强生成（RAG）

把文档切块、向量化入库；提问时检索最相关的几段喂给大模型作答，相当于给 LLM 装上"长期记忆"与"资料库"。这是当下最火的用法。

推荐与多模态搜索

以图搜图、以文搜图、相似商品/内容推荐——把"用户喜欢的东西"向量化，再找它的邻居。

语义去重 / 聚类

近似重复检测、海量内容聚类：靠"向量靠得够近"来判断"内容够像"。

异常检测

正常样本在空间里聚成一团，离群点（向量远离簇）往往就是可疑对象。

Milvus 长什么样：分布式的整体形态

从最高处看，Milvus 把系统拆成三类角色，外加三个外部依赖。这里只建立轮廓，第 2 课画"全景地图"、第 3 课走"一次请求的一生"，会逐步放大每一块：

SDK / 客户端
pymilvus 等
→
Proxy
接入 · 校验 · 路由
→
协调器
root / data / query coord
→
工作节点
data / query / streaming node

这三类角色各司其职：Proxy 是统一入口，负责接收请求、做参数校验与鉴权，再把请求路由到该去的节点； 协调器（RootCoord / DataCoord / QueryCoord）是"大脑"，管元数据、分配任务、调度负载，但自己不亲手搬数据； 真正干活的是工作节点——DataNode 处理写入与落盘，QueryNode 加载段并执行检索，StreamingNode 承载流式日志。 "协调器管调度、节点管执行"是 Milvus 架构里反复出现的主旋律。

底下还垫着三块外部依赖，分别承担"记账、存货、写日志"：etcd 存元数据（有哪些集合、分片在哪台机器）； 对象存储（MinIO / S3）存真正的数据与索引文件；消息队列 / WAL（Pulsar / Kafka / 内置 woodpecker）承载写入日志， 让"写"先落到一条可重放的日志上。Milvus 的一个核心设计就是"日志即数据（log as data）"——这条线我们会在写入链路（第 15 课起）反复用到。

为什么是 Go + C++？这是一种"各取所长"的分工：Go 写分布式控制面——协调、调度、RPC、并发管理， 开发效率高、并发模型顺手；C++ 写计算密集的内核——段内检索（segcore）、索引、距离计算，追求极致性能、贴近硬件（也能调度 GPU）； 再加少量 Rust（tantivy）做全文倒排索引。配合存算分离，计算节点与存储能各自独立扩展——这正是它敢说"弹性伸缩到十亿级"的底气。

三种部署形态：按规模选一种

同一套理念，Milvus 提供从"一个文件"到"一个集群"的三档形态，按数据量和场景选择即可：

形态怎么跑适合规模感
Milvus Litepip install，本地一个文件入门、原型、笔记本上做实验百万级以内
Standalone单机，所有组件在一个进程中小规模、单机部署千万级
ClusterK8s 上分布式多副本生产、高可用、弹性伸缩十亿级及以上

无论哪种形态，对外的 API 与核心概念是一致的：你写的 create_collection / insert / search 代码不用改， 只是底层从"一个进程"长成了"一个集群"。一个常被低估的特性是"边写边查"：刚 insert 进去的新数据， 不必等"建好索引"就能被搜到——Milvus 靠流式日志 + 可查询的"增长段"做到数据实时新鲜，这一点我们在写入链路与查询链路会反复看到。 这正是本教程的视角——我们不教你怎么用（官方文档已经很好），而是带你钻进引擎，看它内部是怎么实现这一切的。

📌 本课要点
  • 向量 / embedding：AI 模型把非结构化数据编码成的一串数字；语义相近 ⇒ 向量相近。
  • ANN 相似检索：在向量空间找最近邻；用专门索引把检索从"逐个比对"加速到毫秒级。Milvus 把它当一等查询。
  • 库 vs 数据库：算法库（FAISS）只算 ANN；Milvus 还补齐持久化、实时更新、混合过滤、扩展、高可用——把检索变成服务。
  • 整体形态：Proxy（接入）+ 协调器 + 工作节点，外加 etcd / 对象存储 / 消息队列三大依赖；核心思想"日志即数据"。
  • 三种部署：Lite（一个文件）→ Standalone（单机）→ Cluster（K8s 集群），API 与概念一致。

Milvus 的源码乍看庞杂，但有一条最省力的切入线索：它用三种语言，各干一种最擅长的活。看懂这条分工，半张地图就已经在脑子里了：

Go · 分布式控制面

占 internal/ 的绝大多数。负责协调、调度、RPC、元数据、并发管理——也就是"把一群机器组织起来"这件事。Go 的并发模型与开发效率，正好适合写控制面。

C++ · 计算内核

集中在 internal/core/。负责段内检索、索引、距离计算、表达式求值这些计算密集的脏活累活，贴近硬件、追求极致性能（也能驱动 GPU）。

第三种语言是少量 Rust：它只负责一块很专的领域——全文倒排索引，藏在 internal/core/thirdparty/tantivy 里（tantivy 是一个 Rust 写的全文检索库）。 所以一句话记牢分工：Go 管"把机器们协调好"，C++ 管"把一段数据搜得飞快"，Rust 管"全文检索那一小块"。 C++ 内核里又细分成几个子模块——segcore（段内检索引擎）、index（索引构建与加载）、query（查询执行）、exec（执行算子）、expr（标量表达式）—— 这些名字会在后面的课里反复出现，现在混个脸熟即可。

举个最直观的例子感受这套分工：当你发起一次 search，请求会同时穿过这三种语言——先由 Go 写的 Proxy 接住、校验、把请求路由到对的 QueryNode； QueryNode（也是 Go）找出该搜哪些段后，把"段内的相似度计算"这一步下沉给 C++ 的 segcore 去飞快地算；如果你的查询里还带了全文检索条件， 那一小段就轮到 Rust 的 tantivy 出场。三种语言像接力赛一样把一次请求跑完——Go 负责"组织流程"，C++ 负责"算得快"，Rust 负责"全文那一棒"。 看懂这一点，你就明白了为什么 Milvus 要"混血"：不是炫技，而是让每段代码都跑在它最擅长的语言上。

把镜头拉近：search 在 Go 与 C++ 之间怎么交接

上一段说三种语言像接力，但那根"棒"到底在哪一行代码交接？以 search 为例，把镜头再拉近一档：当 QueryNode 走到"要在某个段里真正算距离"这一步时，调用的是 Go 侧 internal/querynodev2/segments/segment.go 里的 LocalSegment.Search。这个方法自己并不算距离——它做的是把请求通过 cgo（Go 调 C 的桥）递进 C++。代码里那个名叫 cgoSearch 的计时器，量的就是"这趟在 C++ 里待了多久"，它的存在本身就标出了语言边界的位置。

越过 cgo 这道门，就进了 C++ 的地盘：internal/core/src/segcore/segment_c.cpp 里的 AsyncSearch 接住请求，把它丢进一个专门的搜索线程池（getSearchCPUExecutor）异步执行，并开一个名为 SegCoreSearch 的链路追踪 span。真正的近邻计算就发生在这里；算完，每段的 topK 再沿原路回传给 Go，由 QueryNode 归并。这趟来回让你看清一件事：Go 与 C++ 的边界不是抽象口号，而是一行具体的 cgo 调用——Go 端负责"决定搜哪些段、收集与归并结果"，C++ 端负责"把一段数据搜到极致快"，cgo 就是它们之间那根结实的接力棒。后面查询链路的课会沿着这根棒一路钻进 segcore，这里你只要记住"棒在 LocalSegment.Search 这一行交出去"即可。

顺带点一个工程上的讲究：跨 cgo 这道门本身是有成本的——要在 Go 的协程调度与 C++ 的线程之间来回切换。所以 Milvus 不会"一行一调"，而是以"一个段、一次搜索"为粒度成批跨越，把尽量多的计算攒在 C++ 一侧一次做完。这也解释了 segcore 为什么要把搜索丢进自己的线程池异步执行：让 C++ 安心吃满 CPU 做密集计算，而 Go 这边的协程不必干等、可以腾出手去伺候别的请求。边界划在哪、怎么跨，往往决定一套"混血"系统跑得顺不顺——这正是 Milvus 宁愿把"算得重的那一整段"成块下沉给 C++ 的深层原因。

运行时有哪些组件：先纠正一个常见误解

很多旧资料会画出"三个独立的协调器"——RootCoord、DataCoord、QueryCoord 各是一个进程。这在今天的 Milvus 里已经不准确了。 当前版本把这三个协调角色合并进了同一个进程 MixCoord。也就是说：三个协调"角色"在代码里仍然存在（分别是 internal/rootcoord、 internal/datacoord、internal/querycoordv2 三个模块），但它们被打包成一个 MixCoord 一起部署，不再是三个独立进程。

这是一个"逻辑 vs 物理"的经典区分，务必分清：在逻辑上，元数据管理（Root）、写入与段管理（Data）、查询负载调度（Query）仍是三件不同的事； 但在物理上，它们如今住在同一个进程里。证据就在代码里——internal/types/types.go 中的 MixCoord 接口， 直接把 RootCoordServer、QueryCoordServer、DataCoordServer 三个服务接口嵌在了一起（本课末尾会贴出这段代码）。 所以今天 Milvus 的运行时组件是这六个：

组件语言职责（一句话）internal 包
ProxyGo对外唯一入口：校验、鉴权、分配时间戳、路由、跨分片归并proxy
MixCoordGo三合一协调器：元数据 + 段管理 + 查询调度（含 streamingcoord）coordinator · rootcoord · datacoord · querycoordv2

两个容易踩的坑顺便点破：①没有单独的 indexnode——索引构建是由 DataCoord 调度、交给一个 datanode 工作进程去跑的，并不存在一个叫"索引节点"的独立组件； ②Standalone 与 Cluster 跑的是同一套组件，区别只在"住在几个进程里"。

逻辑一张图，物理两种形态

同样这六个组件，Standalone（单机）把它们塞进一个进程，省心、适合中小规模；Cluster（集群）则把每个组件拆成独立的 K8s Pod， 各自按需扩缩容、互不拖累。注意：对你写的代码而言，两者毫无区别——同样的 create_collection / insert / search，底层从"一个进程"长成"一个集群"，API 与概念完全一致。

Standalone（单机）

MixCoord + Proxy + 三种 Node + CDC 全在一个进程里；依赖（etcd / 对象存储 / MQ）可内嵌或外接。部署简单，适合中小规模与单机生产。

Cluster（集群）

每个组件是独立的 Pod，QueryNode / DataNode 可各自扩到多副本。弹性、高可用，面向十亿级生产。逻辑角色不变，只是"分家"了。

一条贯穿全书的主线：控制面 vs 数据面

如果说"三种语言"是横向的切法，那么"控制面 vs 数据面"就是纵向的切法——这是理解 Milvus 最重要的一根线，几乎每一课都会回到它。 所谓控制面（control plane），管的是"谁该做什么"：集合的结构、分片落在哪台机器、段该不该刷盘、查询该路由到哪个副本—— 这些决策与元数据都归 MixCoord 这个大脑。而数据面（data plane）管的是"真正把数据搬动、计算出来"： 写入的字节、刷盘的 binlog、检索时一条条算出来的距离——这些重活全在工作节点和 C++ 内核里发生。

为什么要这么分？因为这两件事的脾气完全不同。控制面要的是"正确、一致、别出错"——元数据错一点，整个集群就乱套，所以它宁可慢一点、走 etcd 做强一致； 数据面要的是"快、能并行、能水平扩展"——它可以容忍"近似"、可以多副本并发，机器不够就加机器。把这两种诉求拆开， 各自用最合适的技术去满足，正是分布式系统反复验证过的智慧。所以你会看到一句反复出现的口诀："协调器管调度、节点管执行"—— MixCoord 只发号施令、绝不亲手搬一个字节；DataNode / QueryNode 闷头干活、不操心全局怎么编排。

这根主线还顺手解释了"为什么要存算分离"：当数据面的节点不自己保存长期状态（状态都托管给 etcd 和对象存储）时，它们就变成了"用完即弃"的算力—— 查询忙了就多拉几个 QueryNode、写入忙了就多拉几个 DataNode，闲了再缩回去，而数据一个字节都不会丢。计算与存储能各自独立伸缩， 这正是 Milvus 敢承诺"弹性扩展到十亿级"的工程根基。记住这根线，后面的写入链路、查询链路、一致性，都只是它在不同场景下的展开。

三块外部依赖：记账、存货、写日志

Milvus 自己不重复造轮子，而是把三件"基础设施级"的脏活外包给成熟系统。这也是存算分离能成立的前提：

  • etcd · 记账：存元数据——有哪些集合、分片在哪台机器、谁是谁的副本。小而关键，是集群的"户口本"。
  • 对象存储（MinIO / S3）· 存货：存真正的数据与索引文件——刷盘后的 binlog、建好的索引。便宜、海量、持久。
  • 消息队列 / WAL（Pulsar / Kafka / 内置 woodpecker）· 写日志：承载写入日志，让"写"先落到一条可重放的日志上。这正是 Milvus "日志即数据（log as data）"的根基（第 15 课起细讲）。

把这三块垫在底下，计算节点就变得"无状态"了——挂了重启、按需扩容都不丢数据，因为真正的状态都在 etcd 与对象存储里。这就是存算分离能弹性伸缩到十亿级的底气。

这里也顺手说清一个常被问到的点：这三块依赖不是"可有可无的配角"，而是 Milvus 架构的一部分。它们的选择是可插拔的——消息队列既可以用 Pulsar / Kafka 这类成熟中间件， 也可以用 Milvus 内置的 woodpecker（省去额外部署一套 MQ 的麻烦）；对象存储既可以是云上的 S3，也可以是本地自建的 MinIO。Milvus 通过一层抽象把这些差异挡在外面， 让上层逻辑无需关心"日志到底落在哪、文件到底存在哪"。这种"把基础设施抽象掉"的设计，正是它能在云上、私有化、单机之间无缝切换的原因。

把写入想成往邮局寄信：你不会把信直接塞进收件人家门，而是先交到邮局柜台（Proxy），柜台盖一个时间戳（收件时间）、 按地址分拣，再把信登记进一本流水账（WAL 日志）。只要账记上了，这封信就"算寄出了"，哪怕后续的投递慢一点也丢不了。 而查询更像查档案：你说"给我看截至某个时刻为止的全部记录"，档案管理员据此圈定一个时间快照， 保证你看到的是一份前后一致的状态，而不是边查边变的半成品。时间戳，就是这两件事共同的钥匙。这一课，我们就握着这把钥匙，把读写两条链路彻底走通一遍。
🌍 宏观理解
一次请求的主线只有两条：写沿着日志走（Proxy→WAL→growing 段→flush），读靠扇出归约（Proxy→QueryNode 分片→segcore→reduce）；贯穿这两条线的钥匙，是时间戳。

写入的一生：insert 走过的七站

先看写。一条 insert 从 SDK 出发，要走过下面这条链路。请特别留意两个关键动作：在 Proxy 盖时间戳、 以及"先写日志"——这两点是后面理解一致性与崩溃恢复的地基：

1

客户端 / SDK

pymilvus / Go milvusclient 把若干行（向量 + 标量字段）打包，发往 Proxy。

2

Proxy · 校验 + 补主键 + 盖时间戳

校验 schema；若主键是 auto-id 就自动补上；向 TSO 申请一个全局时间戳 ts，盖在这批数据上。

3

Proxy · 按主键哈希分片

用主键把每行哈希到不同的 vchannel（虚拟分片），决定它该进哪条写入流——这样数据天然被打散到多个分片并行处理。

4

追加到 WAL / 流式日志

Proxy 把这批数据追加进 WAL（一条可重放的写入日志）。账一记上，这次写入就算成功返回了。

5

StreamingNode / DataNode · 消费日志

工作节点顺着日志往下读，把刚写入的行放进一个"增长段（growing segment）"——一块还在内存里、可以被立即查询的数据。

6

刷盘成 binlog

当增长段攒到一定大小 / 时间阈值，就刷盘（flush）成binlog，持久化到对象存储，段从此"封存（sealed）"。

7

DataCoord · 记账段元数据

DataCoord（MixCoord 里的 Data 角色）记录"哪些段、在哪、多大、刷没刷"，作为后续查询与压缩的依据。

这里有三个观念值得现在就刻进脑子。其一，"日志即数据（log as data）"：Milvus 把"写"先变成"往日志里追加一条记录"， 日志才是唯一的真相来源，段、索引都只是日志的"物化视图"。这让崩溃恢复变得简单——重放日志即可。其二，"边写边查"： 新数据进了增长段就能被搜到，不必等刷盘、更不必等建索引，这正是 Milvus 数据"实时新鲜"的来源。其三，"一批原子可见"： 同一批 insert 共享同一个时间戳，要么一起可见、要么都不可见，不会让你查到"半批"数据。

有一个地方值得停下来多想一层：为什么写入要"先过日志"，而不是直接写进段、写进对象存储？这看似绕了远路，实则是分布式存储的核心智慧。 直接写段有三个麻烦：一是慢——对象存储适合"大块、批量"写，逐行写又慢又贵；二是难恢复——写到一半进程挂了，段处于"残缺"状态，很难判断哪些行成功了； 三是难并发——多个写入者同时改同一个段，要费很大力气做锁。而"先写一条只追加（append-only）的日志"把这三件事一次解决： 追加是顺序写，飞快；日志天然有序，重放到哪一条一清二楚，崩溃后接着重放即可；多个写入只是往同一条日志尾部排队，无需互相加锁。 段和索引则在"事后"由消费者从日志慢慢物化出来——这就是把"写得快"和"查得快"这对矛盾解耦的关键招式。

崩溃与重放：日志即数据的实战红利

前面说"日志即数据"听起来像口号，它最实在的红利要等到崩溃恢复时才兑现。设想消费日志、攒增长段的那个工作节点突然宕机：内存里还没刷盘的增长段一瞬间全没了。换作"直接写段"的系统，这往往意味着数据丢失；但在 Milvus 里，丢掉的只是"从日志物化出来的中间结果"，而唯一真相——那条 WAL 日志——好端端地躺在消息队列／对象存储里，一个字节都没少。

于是恢复就变成一件机械的事：节点重启后，不必从头重放整条日志，而是从它上次记下的检查点（checkpoint，即消费位点 / seek position）开始，把"检查点之后"的那段日志重新消费一遍（replay），增长段就被原样重建出来，仿佛什么都没发生过。检查点的作用就像看书夹的书签：你不必每次从第一页读起，翻到书签那页接着读即可。正因为日志天然有序、可重放，"读到哪了"永远是一个明确的位置，崩溃后接着读就行——这就是把"写"设计成"带时间戳的、只追加的日志"换来的最大底气。

再往上看一层，这也解释了为什么 Milvus 的工作节点能做到"无状态、用完即弃"：既然任何节点的内存状态都能靠重放日志重建，那么节点挂了就换一个、负载高了就多拉几个，数据安全完全不依赖某一台机器还活着。"日志即数据"与上一课的"存算分离"，本质上是同一个设计哲学的一体两面——把唯一真相托管给可重放的日志与对象存储，计算节点便获得了随意伸缩、随时重建的自由。

还有一个常被忽略的细节值得点一句：检查点不是凭空前进的，而是跟着"刷盘"走。当增长段刷成 binlog、稳稳落进对象存储后，"这段日志已被安全物化"才算坐实，消费位点便可以安全地往前挪——这样崩溃后要重放的日志就越来越短，恢复也越来越快。换句话说，flush 既是为了让历史数据落盘，也是为了给日志"减负"。这条"写日志 → 消费 → 刷盘 → 推进检查点"的循环，正是写入链路在背后悄悄转动的飞轮，第 15 课起我们会逐圈拆解它。值得一提的是，检查点也不必"每条都记"——记得太频繁本身是种开销，记得太稀疏又会拉长崩溃后的重放窗口，多久推进一次是吞吐与恢复速度之间的又一处工程权衡。你会发现，这种"快与稳之间拿捏分寸"的取舍，几乎是贯穿整个 Milvus 设计的母题。

查询的一生：search 走过的六站

再看读。一条 search 的链路和写不同——它要同时问很多个分片，再把结果层层归并成最终的 topK。 关键动作同样有两个：在 Proxy 定一个"保证时间戳"（决定你能看到多新的数据），以及把"段内的相似度计算"下沉给 C++ 内核：

发起 search(向量, topK)
→
Proxy
定保证时间戳 · 建查询计划 · 按分片路由
→
QueryNode · delegator
圈定 封存段 + 增长段
→
segcore (C++)
段内检索 · 每段 topK

把这条链路一站站说清楚：

1

Proxy · 定"保证时间戳"

根据你选的一致性级别推导一个 guarantee timestamp：要"看见最新"（Strong）就取最大时间，要"够快、容忍少量延迟"（Bounded）就取稍早一点的时间。这个 ts 决定了你这次能看到多新的数据。

2

Proxy · 建计划 + 路由

把查询解析成一个搜索计划（向量字段、度量、topK、标量过滤条件），再按分片把请求分发到相关的 QueryNode。

3

delegator · 圈定要搜的段

每个分片有一个代理者 / 分片领导（delegator / shard leader），它清楚本分片有哪些封存段 + 增长段，并据保证时间戳等数据"够新"了再放行搜索。

4

segcore (C++) · 段内检索

真正"算距离"的脏活下沉到 C++ 的 segcore：在每个段内沿索引找近邻，吐出这一段的 topK。

5

节点内归并

一个 QueryNode 上可能有很多段，它先把各段的 topK 归并成"本节点的 topK"，减少要往上传的数据。

6

Proxy · 跨分片归并 → 客户端

Proxy 收齐各分片的 topK，再做一次全局归并，排出最终的 topK 返回客户端。"分头搜、层层并"是检索的标准套路。

注意搜索的"扇出—归约（scatter-gather）"形状：Proxy 把一个请求扇出给多个分片、每个分片再扇给多个段并行算， 然后结果反向归约——段 topK → 节点 topK → 全局 topK。每一层都只往上传"前 K 个"，所以即便底下有上亿条向量， 网络上流动的也只是很少量的候选。这就是分布式检索既快又省的关键。

顺便回答一个常见疑问：为什么读和写要走两条不同的路？因为它们的目标根本不同。写追求"尽快落定、绝不丢失"，所以它走"日志"这条顺序、单向的快车道， 一路向前、不回头；读追求"在海量数据里挑出最像的几条"，所以它必须扇出到所有相关分片和段、并行去算，再把结果归约回来。 一个是"把水灌进管道"，一个是"从万千水库里舀出最甜的那几瓢"——形状自然不同。但请注意，读其实要同时看两种段： 已经刷盘的封存段（历史数据）和还在内存里的增长段（刚写入、尚未落盘的新数据）。正因为 delegator 把这两者都纳入检索范围， Milvus 才能做到"边写边查"——你上一秒 insert 的数据，下一秒就能在增长段里被搜到。这也是为什么"写入的增长段"和"查询的段选择"是同一枚硬币的两面。

时间戳：读写共用的那把钥匙

把上面两条链路并排一看，你会发现时间戳（timestamp）是贯穿读写的同一根轴：写给每批数据盖一个 ts（决定它"何时发生"）； 读用一个保证 ts 圈定一个快照（决定它"看见到何时为止")。两者用的是同一套全局时钟（TSO），于是"先后顺序"在整个集群里有了统一的定义：

写入流ts=10 批Ats=20 批Bts=30 批C…持续追加…
读 (Strong)取 ts=最新看见 A·B·C 全部
读 (Bounded)取 ts=稍早看见 A·B（容忍 C 略迟）

这里只需建立直觉：保证时间戳越"新"，看到的数据越全，但可能要多等一会儿（等分片追上这个 ts）；越"旧"则越快、但可能漏掉最新写入。 这正是一致性级别在背后做的权衡。它的完整机制——TSO 怎么发号、delegator 怎么等、快照怎么保证——留到第 30 课专门展开，这里点到为止。

再补一句关于 TSO（Timestamp Oracle，时间戳分配器）的直觉：它是整个集群的"统一发号机"，保证发出去的时间戳全局单调递增—— 后申请的一定比先申请的大。有了这台"全局时钟"，分布在不同机器上的写入才有了公认的先后次序，读取也才能用一个 ts 干净地切出"截至此刻"的快照。 你可以把它想成银行叫号机：每个人来都撕一张号，号码只增不减，于是"谁先谁后"再无争议。TSO 住在 MixCoord 里（Root 角色），Proxy 每次处理读写都先向它要一个号。 正是这台朴素的发号机，撑起了 Milvus 在分布式环境下的时序一致性——一个看似简单、却无处不在的基础设施。

📌 本课要点
  • 写入链路：SDK → Proxy（校验、补主键、盖时间戳、按 PK 哈希分片）→ 追加 WAL → StreamingNode/DataNode 消费 → 增长段 → 刷盘成 binlog → DataCoord 记账。
  • 日志即数据：先写可重放的日志，写入即返回；段与索引是日志的物化视图，崩溃后重放即可恢复。
  • 查询链路：SDK → Proxy（定保证时间戳、建计划、路由）→ QueryNode delegator 圈定封存段+增长段 → segcore(C++) 段内 topK → 节点归并 → Proxy 跨分片归并 → 客户端。
  • 扇出—归约：每一层只上传"前 K 个"，所以海量向量下网络上只流动极少候选。
  • 时间戳是读写共用的钥匙：写盖 ts、读用保证 ts 取一致快照；一致性级别就是在"看得多新 vs 等得多久"之间权衡（细节见第 30 课）。

向量与相似度

第一部分我们说过，Milvus 把"找相似"变成"在向量空间里找最近邻"。这一课把这句话拆开揉碎： embedding（嵌入）到底是什么、向量空间长什么样、衡量"像不像"的度量（metric）有哪几种、 什么时候该用哪一种，以及为什么精确最近邻太慢、不得不转向 ANN。把这些基础打牢，后面所有关于索引与查询的课才有地基。

🔌 生活类比
把每条数据想成地图上的一个点：经度纬度只有两个数，就能定位一座城市；而 embedding 是用几百上千个数给一段文字、一张图定位。 "两座城市远不远"用直线距离量；"两段话像不像"也一样——只不过尺子从二维换成了几百维。度量就是你手里那把尺子： 有人按"直线距离"量（L2），有人按"方向是否一致"量（余弦），选错尺子，"远近"的含义就全乱了。

embedding：把万物编码成空间里的一个点

embedding 是一个训练好的神经网络模型的输出：喂进一段文字、一张图、一段音频，它吐出一串固定长度的浮点数——这就是向量。 关键性质只有一条，但极其重要：语义相近的输入，产出的向量在空间里也彼此靠近。"猫"和"小猫"几乎重叠，"猫"和"利率"则隔得很远。 于是"语义相似"这件以前说不清、道不明的事，第一次有了一把可计算的尺子。

两条铁律请记牢：①维度由模型决定（常见 384 / 768 / 1024 / 1536 维），同一个集合里的向量必须同维，否则没法比较； ②灌库与查询必须用同一个模型，否则两边向量根本"不在同一个空间"，比距离毫无意义——好比一个人用米、一个人用英尺报身高，数字再接近也不能直接比。

再补一句关于 embedding "好不好"的直觉：一个好的嵌入模型，是在海量数据上训练出来的，它学会的不是"记住每条数据"，而是把"语义"提炼成方向—— 相关的概念指向相近的方向，无关的概念彼此正交（垂直）。所以"用对模型"几乎和"建对索引"一样重要：再快的检索，建立在错位的向量空间上，找回来的也是一堆看似相似、实则无关的结果。 这也是本教程反复强调"灌库与查询同模型"的原因——它是一切相似检索正确性的前提，而非性能问题。

度量：衡量"像不像"的几把尺子

向量是空间里的点，"两条数据像不像"就翻译成"两个点的距离或相似度有多少"。在动手之前，先补一句关于维度的直觉： 维度不是越高越好。维度越高，模型能塞进去的"语义细节"越多，但存储和计算成本也线性上涨，且过高的维度还会带来后面要讲的"维度灾难"。 工程上常见的取舍是：检索质量要求高就用 768 / 1024 维，要省内存、追求吞吐就用 384 维甚至更低，或者对向量做量化压缩（第 5 课展开）。 同一个集合内维度必须一致，这个数字一旦定下来，就跟着这个集合走到底。Milvus 支持的度量名字是固定且大小写敏感的字符串常量， 最常用的三把尺子是 L2、IP、COSINE：

  • L2（欧氏距离）：两点之间的直线距离，越小越像。它关心"绝对位置差多少"，对向量的长度（模长）敏感。
  • IP（内积，Inner Product）：对应元素相乘再相加，越大越像。它同时受"方向是否一致"和"模长大小"影响。
  • COSINE（余弦相似度）：只看两个向量的夹角，完全忽略模长，越大（越接近 1）越像。本质是"先归一化再算内积"。
一次距离计算：查询向量 q 与库内向量 v 逐维相乘再相加，得到内积（IP）；L2 则是逐维相减、平方、求和
q0.900.100.80
v0.880.120.79
IP0.90·0.88+0.10·0.12+0.80·0.79=1.42 越大越像
L2²0.02²+0.02²+0.01²=0.0009 越小越像

这里藏着一个常见困惑："距离"和"相似度"方向相反。L2 是距离，越小越好；IP / COSINE 是相似度，越大越好。 Milvus 在内部会统一处理这种排序方向，但你在选度量、看返回分数时要心里有数：看到 L2 别误以为"分数越高越像"。

归一化：为什么 IP 和 COSINE 常常画等号

"归一化（normalize）"指把向量缩放成长度为 1（除以它的模长），只保留方向、丢掉长度。这件小事会带来一个漂亮的结论： 对归一化后的向量，内积（IP）就等于余弦相似度（COSINE）。因为余弦本来就是"先归一化再算内积"，当向量已经是单位长度时，这一步归一化就免了。

未归一化

向量长度各不相同。用 IP 时，长向量天然占便宜（内积偏大）； 这时若你真正想比的是"方向/语义"，更应该用 COSINE 或 L2。

已归一化（长度=1）

长度信息被抹平，只剩方向。此时 IP == COSINE， 且 L2 与 COSINE 单调对应——用谁结果排序都一样。很多 embedding 模型默认输出就已归一化。

实战经验：如果你的模型文档说"输出已归一化"，那 IP 与 COSINE 等价、且最省事； 如果没归一化、又只关心语义方向，用 COSINE 最稳妥；如果连绝对位置都有意义（比如某些数值特征），才考虑 L2。 最重要的纪律是：建索引和搜索必须用同一种度量，否则"远近"的定义对不上，召回会一塌糊涂。

二值与稀疏向量：另外几把专用尺子

除了常见的浮点稠密向量，还有两类特殊向量，它们配专用的度量：

  • 二值向量（BinaryVector）：每一位非 0 即 1（比如指纹、分子结构哈希）。度量用 HAMMING（有多少位不同）或 JACCARD（两个集合的重叠程度）。
  • 稀疏向量（SparseFloatVector）：维度极高但绝大多数是 0（比如 BM25、SPLADE 这类词项权重）。这类向量天然适合用 IP（内积） 来打分。
度量含义方向典型用途 / 向量类型
L2欧氏直线距离越小越像稠密浮点；关心绝对位置
IP内积越大越像已归一化稠密向量；稀疏向量
COSINE夹角余弦越大越像稠密向量，只比方向/语义

把上面的知识收成一份可操作的决策清单。选度量这件事，归根到底取决于你的向量从哪来、要比什么，三句话就能定：

看模型文档

若写明"输出已归一化（normalized / unit length）"，直接用 IP——它和 COSINE 等价又最省算力。 绝大多数主流文本 embedding（如 bge 系列）都属于这一类。

只比方向/语义

向量没归一化、你又只在乎"语义方向像不像"，用 COSINE 最稳：它自动忽略模长，不会被"长向量"带偏。

剩下两种情况：当绝对位置/数值大小本身有意义（某些工程特征、坐标类数据）时，才用 L2；当向量是二值或稀疏时， 按前面的表走 HAMMING / JACCARD / IP。一个最容易踩的坑再强调一次：同一个集合，建索引用什么度量，搜索就必须用什么度量，两者必须严格一致—— 这不是"建议"，而是"规则"，违反它不会报一个友好的错误，而是悄悄返回一堆错的结果，排查起来非常费劲。

最后给一个收尾的直觉：度量决定"什么叫近"，模型决定"向量好不好"，而索引（下一课）决定"找得快不快"。三者是正交的三件事， 但在一次真实检索里缺一不可。把这三层在脑子里分开，你以后看任何向量检索的配置、调参、排障，都能立刻定位到"问题出在哪一层"。

一个算到底的例子：同两条向量，三把尺子各得什么

把前面的度量真正"算一遍"，直觉会牢固很多。取两条二维向量（维度取小只为方便手算，几百维同理）： a = [3, 4]、b = [4, 3]，它们的模长都是 5（因为 3²+4²=25）。三把尺子分别给出：

COSINEIP ÷ (|a|·|b|) = 24 ÷ 250.96相似度，越接近 1 越像

topK 与分数阈值：一次搜索返回什么

把"度量"用起来，一次搜索就是：给定一个查询向量，在集合里按度量算出每条的"远近"，取最接近的 K 条返回，这就是 topK。 返回结果是按相似度排好序的——第 1 名最像，依次往下。业务里还常配一个分数阈值：把"虽然排在最前、但其实不够像"的结果挡在门外， 避免给用户返回一堆勉强凑数的近邻。注意阈值的方向要跟着度量走：L2 是"小于某个距离才算像"，COSINE / IP 则是"大于某个分数才算像"。

高维空间为什么反直觉

我们对"距离"的直觉来自二维、三维世界，但 embedding 动辄几百上千维，那里的几何和日常经验大不相同。一个著名现象叫"维度灾难"： 维度一高，几乎所有点之间的距离都趋于接近，"最近"和"次近"的差距被抹平，精确比较的区分度下降；同时，要在高维空间里"靠分块覆盖"做精确检索， 所需的格子数随维度指数级爆炸。这两点合起来意味着：在高维下既没法靠简单切分加速精确检索，逐个暴力又太慢——只能退而求其次用近似。

换个角度看，这恰恰解释了为什么 ANN 不是"偷懒"，而是高维世界里唯一可行的工程路线：既然精确的代价是指数或线性爆炸，那么用一点点召回损失， 换来可控的、毫秒级的检索，就成了必然选择。这也是后面所有索引（IVF、HNSW、PQ、DiskANN）共同的出发点——它们只是用不同的招式逼近同一个目标。

ANN 问题：为什么不能"挨个精确算"

有了向量和度量，最朴素的搜索就是暴力检索（brute-force / FLAT）：把查询向量和库里每一条都算一次距离，再排序取最近的 K 条（topK）。 结果绝对精确，逻辑也最简单。问题只有一个字：慢。

算一笔账：1 亿条 768 维向量，一次查询要做约 1 亿 × 768 ≈ 768 亿次乘加运算。单条查询就如此沉重，再叠加成百上千的并发， 任何单机都扛不住。这就是精确最近邻（exact kNN）在大规模下的死穴——复杂度 O(N·D) 随数据量线性增长，数据涨十倍，延迟也涨十倍。

做法怎么找速度结果
精确 kNN（FLAT）和每条都算距离慢 O(N·D)100% 精确
ANN 索引只看一小撮候选快（毫秒级）近似（召回常 95%+）

解法是近似最近邻（ANN, Approximate Nearest Neighbor）：预先把向量组织成专门的索引结构（分桶、建图、压缩……）， 查询时只检查一小撮最可能的候选，而不是全库扫描。代价是偶尔会漏掉个别真正的最近邻——但实践中召回率常在 95%~99% 以上， 而速度能快上几百倍。对绝大多数应用，这是一笔极其划算的买卖："用一点点精度，换巨大的速度"。

这一课只把问题讲清楚：为什么需要 ANN、它在用什么换什么。至于 ANN 具体怎么实现——分桶的 IVF、建图的 HNSW、压缩的 PQ、面向磁盘的 DiskANN—— 是下一课（第 5 课）的主角。把这一课的"度量"和"为什么近似"装进脑子，你就握住了理解所有向量索引的两把钥匙。 顺带说一句：FLAT（暴力检索）在 Milvus 里也是一个真实可选的"索引"——当数据量不大、又要 100% 精确召回时，它反而是最合适的选择。 所以"精确 vs 近似"不是非此即彼，而是 Milvus 留给你的一个按场景权衡的旋钮，这正是下一课要展开的主题。

🌍 宏观理解
一句话串起来：embedding 把数据变成空间里的点 → 度量给出"两点多像"的尺子 → 精确最近邻太慢 → 用 ANN 索引做近似检索。 这四步是 Milvus 一切检索能力的地基。度量名字（L2 / IP / COSINE / HAMMING / JACCARD）和"近似换速度"的取舍，会在后面每一节反复出现。
📌 本课要点
  • embedding：模型把非结构化数据编码成定长向量；语义相近 ⇒ 向量相近。维度由模型定，灌库与查询必须同模型、同维。
  • 度量：L2（距离，越小越像）、IP（内积，越大越像）、COSINE（夹角，越大越像）；建索引与搜索必须用同一种。
  • 归一化：缩放成长度 1 后，IP == COSINE；许多 embedding 默认已归一化。
  • 专用度量：二值向量用 HAMMING / JACCARD；稀疏向量用 IP。名字大小写敏感（见 metric_type.go）。
  • ANN 问题：精确 kNN 是 O(N·D)，规模一大就崩；ANN 用一点点召回损失换几百倍提速。

ANN 算法直觉

上一课我们论证了：大规模下精确最近邻太慢，必须用 ANN（近似最近邻）。这一课就来建立对主流 ANN 索引家族的直觉—— FLAT、IVF、HNSW、PQ、DiskANN各用什么招式逼近"最近邻"，它们在速度、内存、召回这个三角里各站什么位置， 以及最关键的几个调参旋钮（nlist/nprobe、M/ef）到底在调什么。看完你就能给"该选哪个索引"一个有依据的答案。

🔌 生活类比
在一座大图书馆里找"内容最像这本"的几本书。FLAT 是把每一本都翻一遍——最准，但累死人。 IVF 是先按主题分区，只钻进最相关的几个书架找（漏掉别架的风险换来速度）。 HNSW 是顺着"这本像那本"的引用链一路跳过去，几跳就到目标附近。 PQ 是把每本书压成一张"摘要卡"，省地方、翻得快，但摘要丢了细节。 DiskANN 则是把书架搬进仓库（磁盘），只在手边留索引，用得起海量、但每次要去仓库取一趟。

先认清三角：速度 ↔ 内存 ↔ 召回

所有 ANN 索引都在同一个三角里做取舍：查询速度（多快返回）、内存占用（吃多少 RAM）、召回率（找回的 topK 有多接近真正的最近邻）。 这三者不可能同时拉满——你几乎总是在"用一个换另两个"。理解任何一个索引、调任何一个参数，本质都是在这张三角里挪动你的位置。

想更快 / 更省内存

就得接受召回下降：少看候选（IVF 调小 nprobe）、压缩向量（PQ）、把数据放磁盘（DiskANN）。 代价是偶尔漏掉真正的最近邻。

想更高召回

就得付出更多时间或内存：多看候选（调大 nprobe / ef）、保留原始精度（不量化）、把索引全放内存（HNSW）。 极端就是退回 FLAT，100% 召回但最慢。

记住这张三角，下面每个索引家族就都能"对号入座"：它把旋钮拧向了三角的哪个角，又为此放弃了什么。 再强调一点：三角里还藏着一个常被忽略的第四维——建索引的时间与成本。FLAT 几乎不用建（直接存），HNSW 建图很慢、很吃 CPU， IVF 要先跑一遍聚类。线上系统不仅要考虑"查得快不快"，也要考虑"建得起、重建得起"——数据持续写入时，索引是要不断增量或重建的。 这条线我们在第五部分（索引构建与调度）会专门展开，这里先在心里留个位置。

FLAT：不近似的"基准线"

FLAT 不做任何近似，就是上一课说的暴力检索：和每条都算距离、排序取 topK。它的意义有二：①小数据量（几万、几十万）时，它又准又够快，没必要上复杂索引； ②它是衡量其他索引召回率的"标准答案"——其他索引找回的结果有多接近 FLAT，就是它们的召回率。换句话说，FLAT 既是一个能用的索引，也是一把评估别的索引好坏的尺子。 很多人忽略了它的第二个身份，但在调优时它极其有用：没有"标准答案"，你根本无法量化"近似索引到底漏了多少"。

IVF：先分区，再只钻几个区

IVF（倒排文件，Inverted File）先用聚类（k-means）把全库向量分成 nlist 个桶（每个桶一个中心点）。 查询时不再扫全库，而是先找出离查询最近的 nprobe 个桶，只在这几个桶里精确比对。两个旋钮含义很直白： nlist 是"分多少个桶"（建索引时定），nprobe 是"查询时探几个桶"——nprobe 越大，看的候选越多，召回越高但越慢。 打个比方：nlist 像是把仓库分成多少个货区（一次定好），nprobe 像是这次取货你愿意跑几个货区——跑得多自然更不容易漏，但也更费腿。 这两个旋钮一个管"结构"、一个管"每次查询的力度"，配合起来就能把 IVF 精确地摆在三角里你想要的位置。

IVF 还有几个常见变体，区别在"桶里怎么存向量"：IVF_FLAT 桶内仍存原始向量（准）；IVF_SQ8 把每维压成 1 字节（省内存）； IVF_PQ 用乘积量化进一步压缩（更省，但精度更低）。它们共享同一套 nlist/nprobe 直觉，只是在三角里把"内存"这个角拧得松紧不同。

IVF 的一个固有弱点值得知道：它依赖聚类边界。如果某个真正的最近邻不巧落在了"没被探到的桶"里，就会被漏掉——这正是 nprobe 调大能提升召回的原因（探更多桶、漏得更少）。 所以 IVF 的召回对数据分布比较敏感：数据分得均匀时表现好，聚成几个超大簇时就需要更大的 nprobe 才能保证召回。理解了这一点，你调 nprobe 时就不是盲调，而是在"探桶数"和"延迟"之间找平衡。

HNSW：在一张"可导航小世界图"上跳

HNSW（Hierarchical Navigable Small World）不分桶，而是把所有向量连成一张多层图：每个向量是一个节点，和它"邻近"的若干节点连边。 查询时从顶层稀疏图的一个入口出发，每一步都跳到"离查询更近"的邻居，像高速公路换到省道再换到小路，几跳就逼近目标。 三个旋钮：M 是每个节点的连边数（图多密，影响内存与召回），efConstruction 是建图时的候选宽度（建得多细）， ef 是查询时的候选宽度——ef 越大，搜得越广，召回越高但越慢。

1

顶层入口

从最稀疏的一层选一个入口节点，先做"大跨步"逼近。

2

逐层下沉

每到一层，沿边贪心跳向更近的邻居，再下沉到更密的下一层。

3

4

整条路径只访问了全库极小一部分节点，却很可能命中真正的最近邻。

HNSW 是当下最受欢迎的内存索引：召回高、延迟低、对参数不太敏感。代价是内存占用大（图结构 + 原始向量都在 RAM）， 所以它站在三角里"速度高、召回高、但内存贵"的那个角。一个直觉：M 越大，图越密，召回越好但内存和建图时间也越高； ef 是查询时才生效的旋钮，可以在线调——想临时提高召回，把 ef 调大即可，无需重建索引。这种"建图时定结构、查询时调宽度"的设计， 让 HNSW 既能预先建好，又保留了运行时的灵活度。当数据规模大到内存装不下时，就轮到下面两位出场。

PQ 与 DiskANN：当内存装不下

PQ（Product Quantization，乘积量化）是一种压缩手段而非独立索引：它把一个高维向量切成若干子段，每段用一个小码本里的"代表向量"近似， 于是一条向量从几千字节压成几十字节。好处是内存骤降、距离也能在压缩域快速估算；代价是精度损失（用代表向量替代了真实值）。PQ 常和 IVF 组合成 IVF_PQ。

DiskANN则换了个维度省内存：把图索引和向量主要放在 SSD 磁盘上，内存里只留必要的导航信息。它专为单机装不下的超大数据集设计， 用"磁盘的容量"换"内存的紧张"，代价是每次查询要读几次磁盘（靠 SSD 的随机读性能撑住延迟）。一句话：PQ 压数据、DiskANN 挪到磁盘，都是为了在内存有限时仍能服务海量向量。 要注意 DiskANN 强依赖 SSD（固态盘）而非机械盘——它的延迟预算建立在"随机读够快"的前提上，放在机械盘上会慢到不可用。 这也提醒我们：索引选型不只是软件问题，也和硬件（内存大小、盘的类型）深度绑定，这正是把它放进"速度↔内存↔召回"三角去理解的意义。

怎么选：从场景倒推索引

知道了五大家族，真正的问题是"我的场景该选哪个"。别背规则，按三个问题倒推即可：数据量多大？内存够不够？对召回有多苛刻？ 顺着这三问，绝大多数选型就有了答案。

数据小、要绝对精确

几万到几十万条，或调试阶段要"标准答案"：直接 FLAT。它够快、且 100% 召回，省去一切调参烦恼。

内存充足、追求低延迟

百万到千万级、且机器内存放得下：HNSW 几乎是默认首选。高召回、低延迟、对参数宽容，代价是吃内存。

规模大、要可控权衡

需要在速度和召回间灵活拨动：IVF 系列。靠 nprobe 一个旋钮，就能在线调出你要的平衡点。

超大、内存装不下

上亿乃至更多、单机内存放不下：DiskANN 或 IVF_PQ。用磁盘容量或压缩，换"仍能服务"的能力。

这里要破除一个常见误解：没有"最好的索引"，只有"最适合当前三角约束的索引"。同一份数据，在内存富裕的机器上 HNSW 是最优解， 搬到内存吃紧的环境就可能要换成 IVF_PQ 或 DiskANN。所以"选索引"不是一次性决定，而是随你的数据规模、硬件预算、业务对召回的容忍度一起演化的。 Milvus 的价值之一，正是让你换索引而不换业务代码——同一套 create / insert / search，底层索引可以重建替换。

还有一个实战经验：很多团队会先用 FLAT 或 HNSW 跑出一个"召回基准"，再换到更省资源的索引，对照召回率看损失了多少。 因为 FLAT 给的就是"标准答案"，这套"先建基准、再压成本"的方法论，能让你在三角里有据可依地移动，而不是凭感觉调参。 说到底，索引选型是一门工程权衡而非"找最优解"的艺术：先想清楚业务能接受多少召回损失、机器有多少内存预算、查询延迟的红线在哪， 再回到这张三角上对号入座，答案往往就自然浮现了。

这些索引并非 Milvus 自研，而是来自它的向量检索内核库 Knowhere（一个上游/第三方库）。索引名字都是 Knowhere 的 IndexEnum 常量字符串，Milvus 在 C++ 侧引用它们、并按集合配置挑选对应的索引。 有个细节很容易被忽略：这些枚举常量散落在不同文件里，并非集中在某一个文件中定义。下面这段 index/Utils.cpp 列出的，是该文件里确实出现的几个枚举（IVF 与稀疏向量族）—— 请特别注意：本课的两个主角 HNSW 与 DiskANN 并不在这个文件里，它们各自真正的引用位置列在下方表格， 教程不杜撰任何符号的所在文件：

数据模型

前几课都在讲"向量怎么比、怎么搜"。这一课换个角度：数据在 Milvus 里到底是怎么组织的？ 从 collection（集合） 到 schema（模式） 到 field（字段），再到主键、自增、动态字段、分区、分区键， 以及 Milvus 现在支持的全部数据类型——尤其是六种向量类型。把这套"数据模型"理清，你才知道 insert 进去的东西被装进了什么样的结构里。

🔌 生活类比
把 collection 想成一张Excel 表：schema 是表头的设计（有哪些列、每列什么类型），field 是一列，一行就是一条实体（entity）。 主键是那一列"工号"，全表唯一、用来精确定位某一行。分区像是把这张大表按"部门"拆成几个 sheet，查某部门时只翻对应 sheet。 只不过 Milvus 的表里，除了普通的数字、文本列，还能有一种特殊的列——向量列，专门用来做相似检索。

从 collection 到 field：三层结构

Milvus 里数据组织的最外层是 collection，约等于关系数据库里的表（table）。一个 collection 由一份 schema 定义， schema 里列出若干 field（字段），每个 field 规定了名字、数据类型、以及一些类型参数。往 collection 里写入的每一条数据叫一个 entity（实体）， 对应关系库里的一行。这套对应关系记牢，后面所有概念都挂在它上面。和关系库相比，最大的不同只有一处：Milvus 的"行"里可以有一种特殊的"列"——向量列， 它就是做相似检索的那一列。其余的标量列（数字、文本、布尔），作用和关系库一模一样，主要用来做过滤（"价格小于 500""地区等于华东"）。 也就是说，一条 entity 通常长这样：一个主键 + 一个（或多个）向量 + 若干标量字段。检索时既按向量找相似，又能用标量字段加条件，这正是"向量数据库"区别于"纯算法库"的地方。

一条 entity 的解剖：主键唯一定位 + 向量字段做相似检索 + 若干标量字段做过滤，三类字段拼成"一行"

这张"解剖图"几乎就是 Milvus 数据模型的全部精华：一行里同时住着"用来精确定位的主键"、"用来模糊找相似的向量"、"用来精确过滤的标量"。 三者各司其职，缺一不可——没有主键就无法删改去重，没有向量就退化成普通数据库，没有标量过滤就只能做"纯相似"而无法表达业务条件。 把这三类字段在脑子里分清楚，后面读写入与查询链路时，你就能一眼看出每个字段在每一步里扮演什么角色。

这里有一个和关系库的关键差异要先点明：schema 在建集合时定下来后，结构基本是稳定的。维度、主键、度量这些核心元信息一旦确定， 就不能随意更改——这和关系库里随手 ALTER TABLE 加列删列的灵活度不同。原因在于 Milvus 为了极致的检索性能，会围绕 schema 预先规划好底层的列式存储和索引布局， schema 是这套布局的"地基"。所以设计 schema 时要多想一步：哪些字段要参与过滤、哪个字段适合做分区键、向量维度选多少——这些决定会跟着集合走很久。 当然，Milvus 也留了"活口"（下文的动态字段），但核心结构的稳定性，是它换取高性能的必要代价。

顺带一提，collection 之上还有 database（数据库）这一层：一个 Milvus 实例里可以有多个 database， 每个 database 下再放多个 collection，用来做更粗粒度的隔离（不同业务线、不同环境）。本课聚焦 collection 及其内部结构，database 这一层知道它存在即可。

主键、自增与 type_params

每个 collection 必须有一个主键字段（primary key），类型只能是 Int64 或 VarChar。主键全集合唯一， 用来精确定位、删除、去重一条 entity——这一点和关系库的主键完全一致。Milvus 内部很多机制（删除标记、去重、按主键路由分片）都依赖主键， 所以它不是可有可无的装饰，而是整个数据模型的支点：没有唯一主键，"删掉那一条""更新那一条"就无从谈起。

主键可以自增（auto_id）：开启后，你 insert 时不用自己提供主键值，Milvus 会自动分配一个全局唯一的 Int64。 这对"我只关心向量和内容、不在乎 id 是几"的场景很方便；反之，若你的数据本身带业务 id（比如商品编号、文档 UUID），就关掉 auto_id、自己提供主键， 这样你才能用业务侧已知的 id 去精确删改某一条。两种模式各有适用：auto_id 省心，自带 id 可控，按你是否需要"用外部 id 反查"来决定。

type_params（类型参数）是字段类型的补充约束，最常见的两个：向量字段必须给 dim（维度）， VarChar 字段必须给 max_length（最大长度）。它们是建集合时就定死、跟着字段走的元信息——比如"这是一个 768 维的 FloatVector 列"。 为什么非给不可？因为 Milvus 要据此预先规划存储与索引的内存布局：知道每条向量占多少字节、每条字符串最长多少，才能高效地按列存放、批量计算。 这也解释了为什么维度一旦定下就不能改——它早已渗进底层存储的每一个角落。换句话说，dim 和 max_length 不是"软约束"， 而是 Milvus 据以分配内存、组织列存、构建索引的硬性前提；填错了不会"凑合能用"，而是从一开始就建不出正确的存储结构。

动态字段：给 schema 留一道活口

传统关系库的表是强 schema：没在表头声明的列，根本写不进去。但真实业务里，常常有一些临时的、不固定的附加属性，事先没法都列全—— 比如不同来源的商品带着各不相同的标签字段，今天加一个"促销标记"，明天加一个"产地"。如果每加一个属性都要改 schema、重建集合，迭代就太重了。 Milvus 的 动态字段（dynamic field）就是为此设计的：开启 enable_dynamic_field 后，你 insert 时可以夹带 schema 里没声明的键值对， 它们会被统一收进一个名为 $meta 的隐藏 JSON 字段里，事后还能在过滤表达式里使用。这相当于在强 schema 的严谨之外，额外开了一个"什么都能塞"的口袋。

固定字段（声明过的）

schema 里明确定义的 field：类型严格、可建索引、过滤高效。比如 id / vector / price。

动态字段（没声明的）

开启 enable_dynamic_field 后随手夹带的键值，落进 $meta JSON。灵活，但不如固定字段高效。适合放临时、稀疏的属性。

一句话权衡：固定字段图的是"严谨与高效"，动态字段图的是"灵活与免改表"。把高频、要过滤的属性放进固定字段；把零散、临时的属性丢给动态字段，是常见的实践。 要提醒的是，动态字段虽然方便，但它的值存在 JSON 里，过滤时要解析 JSON，效率不如直接对固定字段建立的结构化存储。所以它适合"偶尔用一下"的属性， 而不是"每次查询都要过滤"的核心字段——后者还是老老实实声明成固定 field 更划算。动态字段的真正价值，是让 schema 不必在第一天就想全所有字段， 给快速迭代的业务留了一道喘息的活口。

分区与分区键：把大集合切小

当一个 collection 很大时，把它整体当一坨来查既慢又浪费。分区（partition）是 collection 内部的逻辑切分：你可以按某种规则把数据分到不同分区， 查询时只在相关分区里找，从而剪掉大量无关数据。最典型的用法是按"租户、地区、时间"分区——查某个租户的数据，就只扫它那个分区。 要注意分区是逻辑概念，它和后面要讲的 segment（物理存储单元）是两个层次：一个分区内部，数据仍然会被切成很多个 segment 来存储和索引。 你可以把分区理解成"逻辑上的抽屉"，segment 是"抽屉里实际装东西的盒子"——下一课会专门讲 segment。

手动管理分区很繁琐，于是有了分区键（partition key）：你把某个标量字段（比如 user_id）指定为分区键， Milvus 就会自动根据这个字段的值把 entity 散列到不同分区里。查询时只要在过滤条件里带上这个字段，Milvus 就能自动只搜对应分区， 你完全不用手动建分区、选分区。这在多租户场景里极其好用：一个 collection 装很多用户的数据，靠分区键天然隔离、又互不打扰。

分区和上一课的"索引选型"其实是同一种思想的两次应用：都在想办法"少看一点数据"。索引是在向量层面剪枝（只看近的候选），分区是在数据层面剪枝（只看相关的那一块）。 两者叠加，效果是乘法级的：先用分区键把"几亿条全集合"缩到"几百万条某租户的数据"，再让 ANN 索引在这几百万条里快速找最近邻。理解了这一点，你设计 collection 时就会有意识地问自己： "我的查询几乎总会带哪个过滤条件？"——那个字段，往往就是最好的分区键候选。一个小提醒：分区键一旦在建集合时指定就固定下来， 所以这又是一个需要在 schema 设计阶段就想清楚的决定，和前面说的"维度、主键"一样，属于"开局定终身"的那类选择。

数据类型：标量与六种向量

schema 里每个字段的类型，来自 Milvus 的 DataType 枚举。标量字段负责承载结构化信息、支撑过滤；向量字段承载 embedding、支撑相似检索。 这里有一个极其重要的纠错点：早年的开发者文档里只列了 BinaryVector / FloatVector 两种向量，那份清单早已过时。 以当前代码为准（client/entity/field.go 的 FieldType），Milvus 现在支持的类型如下——标量一大类，向量足足六种：

标量类型里有几个值得单独点一句：VarChar 是变长字符串，必须给 max_length，常用来存原文、标题、URL； JSON 能存一整块嵌套结构，适合放半结构化的元信息；Array 是同类型元素的数组（如多个标签）。 这些标量字段不只是"附带存着"，它们能直接参与过滤表达式——这正是 Milvus 把"向量检索 + 标量过滤"揉在一次查询里的基础。 有了它们，你才能写出"向量最像、且 price < 500、且 tags 包含'新品'"这种混合查询。

这么多向量类型，本质都在三角里做精度 vs 内存的取舍：FloatVector 最精确也最占空间；Float16 / BFloat16 / Int8 是用更少的位数换内存（每条向量占用减半甚至更多）； SparseFloatVector 则专为"高维但绝大多数是 0"的场景（如关键词权重）服务，只存非零项，省得多。还有 BinaryVector，每一维只用 1 个比特表示，配合上一课的 HAMMING/JACCARD 距离，特别适合指纹、去重这类二值特征。选哪种向量类型，和上一课选哪种索引一样，是一道工程权衡题。

把这一课串成一个具体例子收尾：假设要做一个"商品图片检索"的集合，schema 大概是——主键 item_id（Int64，开 auto_id）、 向量字段 image_vec（FloatVector，dim=512）、标量字段 price（Float）、category（VarChar，作分区键）、 再开 enable_dynamic_field 以备临时打标签。建好后，一条"找和这张鞋子图最像、价格小于 500、且属于鞋类"的查询， 就是向量检索（image_vec）+ 标量过滤（price < 500）+ 分区剪枝（category=鞋）三件事一次做完。 你看，这一课讲的每个概念——主键、向量字段、标量字段、分区键、动态字段——都在这一条真实查询里各就各位。把数据模型理清，后面看写入与查询链路时，你才知道数据被装进了什么形状的容器里。

🌍 宏观理解
一句话：collection（表）由 schema 定义，schema 列出 field（列），每条数据是 entity（行）。 主键唯一定位、可自增；动态字段给 schema 留活口；分区/分区键把大集合切小、加速查询，是多租户场景的利器。 字段类型来自 DataType——标量一大类、向量六种（务必以当前代码为准，别用过时的两种向量清单）。 把数据模型想成"带向量列的表"，你就抓住了它的精髓。

Segment 与日志即数据

上一课我们说，一个分区里的数据"仍会被切成很多 segment 来存"。这一课就把 segment（段）——Milvus 里数据与索引的基本单位——讲透： 它的生命周期（Growing → Sealed → Flushing → Flushed →（Dropped）），向量数据怎么从一条vchannel 流进来、又怎么映射到底层的 pchannel， 以及 Milvus 最核心的一条设计哲学——"日志即数据（log as data）"。理解了 segment，你才真正看懂"边写边查"是怎么实现的。

🔌 生活类比
把写入想成一家工厂的流水线。新来的货先堆在正在装的集装箱里（Growing 段，还能继续往里塞）； 装满了就封箱（Sealed，不再接收新货）；封好的箱子被送进仓库归档（Flushing → Flushed，落到对象存储）； 哪天这批货过期了，整箱报废处理（Dropped）。而真正的"账本"不是集装箱本身，而是流水线入口那条不断记录"谁、什么时候、来了什么货"的传送带日志—— 只要日志在，任何时候都能照着它把货重新堆一遍。这条日志，就是 Milvus 的灵魂。

segment：数据与索引的基本单位

在正式拆解之前先建立一个直觉：你在第一部分见过的"边写边查""分布式扩展""崩溃不丢数据"这些能力，追根溯源全都落在 segment 和它背后的日志上。 所以这一课虽然在"前置基础"部分，却是后面所有写入链路、查询链路、压缩与索引章节的共同地基——值得多花点时间彻底吃透。

一个 collection 可能有几亿条数据，Milvus 不会把它当成"一个巨大的整体"来管理，而是切成许多segment（段）。 每个 segment 是一批 entity 的集合（通常几十万到上百万条），它是 Milvus 里存储、刷盘、建索引、加载、检索的基本单位—— 几乎所有数据操作都以 segment 为颗粒度。为什么要切段？因为只有把"大数据"拆成"许多小块"，才能并行处理、独立索引、按需加载、增量更新—— 这正是分布式系统横向扩展的前提。打个比方，与其管理一座没有隔间的巨型仓库，不如把货物装进许多标准集装箱：每个箱子可以单独搬运、单独上架、单独盘点， 整体的弹性和并行度都来自这种"切块"。Milvus 的 segment 就是这样的标准集装箱。每个段还带着自己的元信息（段 ID、所属 collection/partition、行数、状态等），由 DataCoord 统一登记和调度。

segment 分两大类，对应"正在写"和"已写完"两种状态：增长段（Growing）在内存里，正通过流式日志持续接收新写入； 封存段（Sealed）已经写满、变成不可变（immutable），会被刷到对象存储、并在其上构建索引。 这个"可变的增长段 + 不可变的封存段"的二分，是后面理解"边写边查"和"索引构建"的关键。

为什么要强调"不可变"？因为不可变是分布式系统的好朋友：一个段一旦封存就永不更改，那么它就可以被安全地缓存、复制、并行加载到多个 QueryNode， 不用担心一边读一边被改的并发问题。删除和更新怎么办？Milvus 不在原段上改，而是另写一份"删除标记（delete log）"，检索时把被删的主键过滤掉； 真正的物理清理留给后台的压缩（compaction）。所以"不可变段 + 删除日志 + 压缩"这套组合，是 Milvus 在保证高并发读取的同时，又能支持增删改的巧妙设计。

增长段 Growing

在内存里，可变，正在通过 vchannel 持续吸收新数据。它立即可查（保证新数据的新鲜度），但还没建正式索引，靠暴力或临时结构检索。

封存段 Sealed

已写满、不可变，刷盘成 binlog 存进对象存储，并在其上构建 ANN 索引。检索高效，是大部分历史数据的归宿。

生命周期：从 Growing 到 Dropped

一个 segment 会沿着一条固定的状态链演化。这些状态名不是教程瞎编的，而是 Milvus 代码里 SegmentState 枚举里逐字定义的： Growing（增长）→ Sealed（封存）→ Flushing（刷盘中）→ Flushed（已刷盘）→（Dropped 丢弃）。每一步都有明确的触发条件：

Growing内存中 · 可变持续吸收 vchannel 来的新数据，立即可查
Sealed写满 / 超时触发封存不再接收新写入，变不可变
Flushing正在落盘把内存数据写成 binlog 文件
Flushed已存入对象存储可在其上建索引
Dropped被压缩/删除替代逻辑丢弃，等待 GC 回收

解释几个关键转换：Growing → Sealed（封存）由"段写满了（达到大小上限）"或"攒够时间了（超时）"触发——一旦封存，这个段就不再接收任何新写入，变成只读快照。 Sealed → Flushing → Flushed（刷盘）是把内存里的数据持久化成 binlog 文件、写进对象存储的过程；Flushed 之后数据就不怕进程崩溃了， 而且可以在它上面安心建 ANN 索引。Dropped（丢弃）通常发生在压缩（compaction）之后：多个小段被合并成一个大段，或带删除标记的数据被清理，旧段就被标记 Dropped、等待垃圾回收。

1

触发封存

段写满（达到大小上限）或超时——DataCoord 下令把 Growing 段转 Sealed，从此只读。

2

刷盘成 binlog

把内存里按列组织的数据序列化成 binlog 文件，逐字段写入对象存储；写完即 Flushed，数据持久化、不怕崩溃。

3

异步建索引

IndexNode 读取 binlog，在其上构建 ANN 索引（IVF/HNSW…），索引文件也存进对象存储。

4

加载检索

QueryNode 加载封存段及其索引到内存，对外提供高效 ANN 检索；同时 Growing 段补上最新数据。

注意第 3、4 步是异步发生的：刷盘和建索引在后台进行，不阻塞写入，也不阻塞对 Growing 段的查询。这种"写入、刷盘、建索引、加载各自独立、互不阻塞"的流水线， 正是 Milvus 把吞吐做高、把延迟做低的关键。每一环都能独立扩展——写得快了就加 DataNode，建索引慢了就加 IndexNode，查得多了就加 QueryNode。

另外还有几个相关状态值得一提：枚举里还有 Importing（批量导入时的临时段）和 NotExist 等。 但对理解主流程，记住 Growing → Sealed → Flushing → Flushed →（Dropped）这条主线就够了——它贯穿了一条数据"从内存可变、到落盘不可变、到最终被回收"的一生。

vchannel → pchannel：数据从哪条管子流进来

新写入的数据不是直接"塞进" segment 的，而是先经过一条流式日志的管道。这里有两个概念：vchannel（虚拟通道）和 pchannel（物理通道）。 一个 collection 在创建时会被分成 N 个 vchannel（虚拟分片，shard）——这就是"分片数"。每条写入按主键哈希落到某个 vchannel 上， 同一个 vchannel 的数据由一个 DataNode/StreamingNode 负责消费、攒成 Growing 段。vchannel 是逻辑上的"分片"概念。

为什么要分片？因为一个 collection 的写入量可能很大，单条管道扛不住。把它拆成 N 条 vchannel，每条独立消费、独立攒段，就能并行吃下写入—— 这是写入侧横向扩展的关键。分片数在建表时定下，决定了写入的最大并行度，所以是个需要提前规划的容量参数。

而 pchannel 是物理通道——它对应消息队列（WAL）里一个真实的 topic。多个 vchannel 可以共享同一个 pchannel： vchannel 的名字其实就是在 pchannel 名字后面拼上 collection ID 和分片序号。下面这段代码就揭示了二者的映射关系——从一个 vchannel 名字里截掉最后一段，就还原出它所在的 pchannel：

pkg/util/funcutil/func.goToPhysicalChannel / GetVirtualChannel
// vchannel 名 = pchannel 名 + "_" + collectionID + "v" + 分片号
func GetVirtualChannel(pchannel string, collectionID int64, idx int) string {
    return fmt.Sprintf("%s_%dv%d", pchannel, collectionID, idx)
}

// 反过来：从 vchannel 名截掉最后一段，就得到 pchannel
func ToPhysicalChannel(vchannel string) string {
    index := strings.LastIndex(vchannel, "_")
    return vchannel[:index]   // 多个 vchannel 可共享一个 pchannel
}

为什么要分这两层？因为逻辑分片数（vchannel）和物理资源数（pchannel/topic）需要解耦：你可以有很多逻辑分片来并行处理， 但底层物理 topic 的数量受 MQ 资源限制，于是让多个 vchannel 复用少量 pchannel。这是"逻辑与物理解耦"的经典设计——和上一课"分区是逻辑、segment 是物理"异曲同工。

日志即数据：Milvus 的核心哲学

现在来讲 Milvus 最重要的一条设计思想——"日志即数据（log as data）"，也叫"log as a source of truth"。它的意思是： 写入这件事的"真相"，不是某个表、某个段里的数据，而是那条按时间戳排好序、只追加（append-only）、可重放（replayable）的日志。

传统数据库里，"日志"（WAL）只是为了崩溃恢复的辅助，真正的数据在表里。Milvus 把这个关系倒了过来：日志本身就是权威数据源， 其他一切数据形态——内存里的 Growing 段、对象存储里的 binlog、QueryNode 加载的索引——都是对这条日志的"物化（materialized）视图"， 它们靠不断消费、重放这条日志来追上最新状态。一条写入只要成功写进了日志，就算"落地"了，哪怕此刻还没刷盘、还没建索引。

这里还藏着一个细节：日志里的每条记录都带一个时间戳（timestamp）。Milvus 用一套全局时间戳机制给所有写入和查询排序， 于是"在某个时间点之前的数据是什么样"这件事是确定可重现的。这就是后面 MVCC、一致性级别、时间旅行查询的根基——而它们全都源自"日志即数据"这一条。

列式布局：段内部怎么存

最后看一眼 segment 内部是怎么存数据的。Milvus 用列式（columnar）布局：同一个字段的所有值连续存在一起， 而不是像关系库那样一行行存。比如一个段里有 id、vector、price 三个字段，它会把所有 id 存一块、所有 vector 存一块、所有 price 存一块。 落盘后，每个字段对应一份 binlog 文件。

为什么用列式？因为向量检索的访问模式是"把某一列（向量列）整批拿出来算距离"，列式布局让这种批量、连续的访问极其高效（缓存友好、可向量化计算）； 标量过滤也类似——只需扫描相关的那几列，不必把整行读出来。列式 + 按段切分 + 日志驱动，这三件事合起来，就构成了 Milvus 存储层的骨架。

这也解释了为什么上一课讲的那些数据类型会按字段分开存：每个字段（无论是 Int64 主键、FloatVector、还是 JSON）在段里都是独立的一列、独立的一份 binlog。 向量字段那一列交给 ANN 索引去加速，标量字段那几列交给标量索引或暴力过滤——两者在检索时配合完成"先按标量过滤、再在候选里做向量召回"的混合查询。 段、列、日志、索引这几个概念到这里就彻底闭环了。

查询时：Growing 与 Sealed 都会被搜到

很多人有个误解，以为"只有落了盘、建了索引的段才会被搜索"。其实恰恰相反：一次查询会同时扇出到两类段—— QueryNode 上加载的 Sealed 段（历史数据、带 ANN 索引，走高效检索）和仍在内存里的 Growing 段（最新数据、还没建正式索引，走暴力/临时检索）， 最后把两边的 topK 候选归并成一个全局结果返回。少了任何一边结果都不完整：只搜 Sealed 会漏掉刚写入的最新数据，只搜 Growing 则看不到海量历史。

Growing 段（最新）

内存中、可变、还没正式索引。负责新鲜度：刚写进来的数据就在这里，被暴力或临时结构检索。数据量小，暴力也够快。

Sealed 段（历史）

已落盘、带 ANN 索引、被 QueryNode 加载。负责规模与效率：海量历史数据的高效近邻检索都在这里完成。

这套"两类段一起搜、再归并"的设计，正是 Milvus 能兼顾"查得到最新"和"查得快历史"的关键。它也解释了"刚 insert 完马上就能搜到"的原理—— 那条数据此刻就躺在某个 Growing 段里，根本不必等刷盘与建索引。后面查询链路的课会展开"结果怎么归并、一致性级别怎么影响可见性"，这里先记住一句话：Growing 与 Sealed 是查询的两条腿，缺一不可。

依赖与部署形态

前面几课讲的都是 Milvus "内部"怎么组织数据；这一课收个尾，看看 Milvus "外部"依赖什么、又能以哪些形态部署。 Milvus 把"存元数据""存大块数据""记写入日志"这三件事分别外包给三类外部组件：etcd（元数据 + 服务发现）、 对象存储（MinIO/S3，存 binlog 与索引）、消息队列 / WAL（rocksmq / Pulsar / Kafka / Woodpecker）。 再配合三种部署形态——Milvus Lite / Standalone / Cluster——你就能从"能跑起来"一路理解到"能扛生产"。

🔌 生活类比
把 Milvus 想成一家大型连锁餐厅。它不自己造发电机、不自己挖水井，而是接入城市的基础设施： etcd 像是工商登记与门店通讯录（谁在营业、各店地址、统一调度）；对象存储像是城郊的中央仓库（海量食材长期囤放，便宜耐放）； 消息队列像是后厨的点单传送带（每一单按顺序记下、依次处理，断电也能照单复原）。 而"开一家路边小摊（Lite）""开一家单店（Standalone）""开一片连锁（Cluster）"，则是同一套手艺的三种规模。把这三样基础设施和三种规模想清楚，你就掌握了 Milvus 在真实世界里"长什么样、怎么落地"。

为什么要有外部依赖

一个看似简单的问题：Milvus 为什么不把元数据、数据、日志全自己存，非要依赖外部组件？答案是"专业的事交给专业的系统"。 存元数据需要强一致、可监听变更——这正是 etcd 的看家本领；存海量向量和索引需要便宜、近乎无限、高可用的容量——这正是对象存储（S3 类）的强项； 记写入日志需要有序、持久、可重放——这正是消息队列的本职。Milvus 把这三类"难且通用"的能力外包出去，自己专注做"向量检索"这件最核心的事。 如果什么都自己造，不仅要重复造许多成熟的轮子，还要为每一个轮子的可靠性、扩展性单独操心——这显然不划算。更何况 etcd、S3、Kafka 这些系统都已被业界千锤百炼、生态成熟，直接复用它们，既省力又稳妥。

这种"存储与计算分离"的架构还带来一个巨大好处：Milvus 自己的各个节点都变成了无状态（stateless）或近乎无状态的—— 真正的"状态"都沉淀在这三个外部系统里。于是计算节点可以随意增删、崩了重启、弹性伸缩，数据却安然无恙。这是云原生时代数据库的标准做法，也是 Milvus 能横向扩展的根基。 换句话说，当查询压力上来时，你只要多加几个无状态的 QueryNode，它们各自从对象存储拉数据、从 etcd 读拓扑、从 MQ 消费日志，就能立刻分担负载； 某个节点挂了，它身上没有独占的状态，重启或换一台机器重新加载即可，不会丢任何数据。这正是"存算分离 + 外部依赖"换来的弹性与可靠。

① etcd —— 元数据与服务发现

etcd 是一个强一致的分布式键值存储。Milvus 用它存放所有元数据：有哪些 collection、它们的 schema、每个 segment 的状态和位置、索引的元信息等； 同时用它做服务发现——各个节点启动后在 etcd 里注册自己，协调者通过 etcd 知道"现在有哪些节点活着、在哪"。元数据量不大但极其关键，必须强一致、不能丢，一旦元数据出错或丢失，整个集群就会"找不着北"， 这正是 etcd（Kubernetes 也用它）擅长的。它体量虽小，却是整个集群的"户口本"与"通讯录"。配置见 configs/milvus.yaml 的 etcd 段。

为什么元数据要单独拎出来、还非得强一致？设想两个协调者同时改同一个 segment 的状态、或者两个节点都以为自己是某个角色的主——如果元数据存储允许"读到旧值"， 整个集群就会陷入脑裂、状态错乱。etcd 用 Raft 协议保证"任何时刻所有人读到的都是同一份最新值"，并且支持 watch（监听变更）： 某个 key 一变，关心它的组件立刻收到通知。这种"强一致 + 可监听"的组合，正是协调者调度全局、节点感知拓扑变化的底座。所以 etcd 虽然存的数据不多，却是整个集群的大脑中枢，一旦它不可用，集群就无法做任何元数据变更。

② 对象存储 —— 海量数据的归宿

上一课的 binlog、索引文件，最终都存进对象存储：本地开发用 MinIO，云上用 S3 / GCS / 阿里云 OSS 等任何兼容 S3 API 的服务。 对象存储的特点是容量近乎无限、单位成本极低、持久可靠，特别适合存放向量这种"量大、写少读多、要长期留存"的数据。 Milvus 把"重数据"全压到这里，自己只在内存/本地盘缓存热数据。配置见 milvus.yaml 的 minio 段（地址、bucket、AK/SK 等）。

具体来说，对象存储里躺着三类东西：binlog（封存段刷下来的列式数据，每个字段一份）、索引文件（IVF/HNSW 等在 binlog 上建好的 ANN 索引）、 以及 delta 日志（记录删除标记的增量文件）。它们都是不可变的——写一次、读多次、永不原地改，这与对象存储"适合一次写入多次读取"的特性完美契合。 QueryNode 检索时把需要的 binlog 和索引从对象存储按需拉到本地缓存（就是 milvus.yaml 里的 localStorage.path），避免每次都远程读取。 这样一来，对象存储负责"廉价地存全量"，本地盘和内存负责"快速地读热点"，冷热分层、各司其职。值得强调的是，正因为这些文件都不可变，它们可以被任意多个节点同时安全读取、 可以被放心缓存而不必担心失效，这又一次印证了第七课"不可变是分布式系统好朋友"的论断——存储层的简单与可靠，很大程度上就来自这份不可变性。

③ 消息队列 / WAL —— 写入日志的载体

第七课的"日志即数据"那条append-only、可重放的日志，物理上就承载在消息队列（MQ）里——也就是 WAL（预写日志）。 Milvus 支持四种 MQ：rocksmq（基于 RocksDB 的内置轻量队列）、Pulsar、Kafka，以及新一代内置 WAL Woodpecker。 它们都提供"有序、持久、可被多消费者重放"的能力，正是 Growing 段消费、崩溃恢复、读写解耦的物理基础。换个角度看，MQ 在这里扮演的不是"传统消息中间件"的角色，而是 Milvus 的写入主干道——所有写入先进 MQ、再被各路消费者（攒段的、建索引的、做副本的）按需取用，谁快谁慢互不干扰。这正是为什么 MQ 的选择如此重要。

关于"默认用哪个"，milvus.yaml 的注释给了精确的规则（不要凭印象记）：当 mq.type 设为 default 时， 按当前部署模式、依下面的优先级自动挑选——

部署模式默认 MQ优先级顺序说明
Standalone（单机/本地）RocksMQrocksmq（默认）> Pulsar > Kafka > Woodpecker内置、零外部依赖，开箱即用
Cluster（集群）PulsarPulsar（默认）> Kafka > WoodpeckerRocksMQ 在集群模式不支持

两条必须记牢的事实：① Standalone 默认是 RocksMQ，因为它内置在进程里、不需要额外部署，最适合单机；② Cluster 默认是 Pulsar， 且 RocksMQ 在集群模式下不被支持（它是单机内置的，无法跨节点共享）。另外，注释还特别建议：新实例推荐显式使用 Woodpecker， 以获得更好的性能、更简单的运维和更低的成本——它是 Milvus 主推的下一代内置 WAL。mq.type 的合法取值为 [default, pulsar, kafka, rocksmq, woodpecker]。

为什么这套优先级要设计得这么细？因为它要同时照顾"开箱即用"和"向后兼容"两件事：对单机用户，内置的 RocksMQ 不需要你额外部署任何东西，装上就能跑； 对集群用户，RocksMQ 这种单机内置队列根本无法跨节点共享，所以必须退而求其次用 Pulsar/Kafka 这类真正的分布式 MQ。而之所以把 Woodpecker 排在优先级末尾却又"强烈推荐新实例使用"， 是因为优先级要保证老实例升级后行为不变（不能突然把别人的默认队列换掉），但对全新部署，官方希望你主动选 Woodpecker——它作为新一代内置 WAL，在性能、运维成本上都更优。 看懂这套规则背后的"既要兼容老的、又要引导用新的"取舍，你就理解了一个成熟系统是怎么演进默认值的。

三种部署形态

同一套 Milvus，能以三种规模落地，对应不同的使用场景。理解它们的差异，就知道什么时候该用哪一个：

Milvus Lite

一个 pip 就装好的进程内（in-process）版本，数据存在本地文件，无需任何外部依赖。适合学习、原型、笔记本上的小实验、单元测试。能跑通完整 API，但不为大规模/高并发设计。

Standalone

单进程把所有组件打包在一起运行，MQ 默认用内置 RocksMQ，元数据用内嵌或本地 etcd，数据用本地 MinIO。适合中小规模生产、单机部署，运维简单。

Cluster

各组件（协调者 + 各类节点）在 Kubernetes 上独立部署、独立伸缩，MQ 默认用 Pulsar，依赖外部 etcd / S3 / Pulsar 集群。适合大规模、高并发、高可用生产。

形态怎么跑用到的依赖规模何时用

这张表把三种形态的差异一次摆清：从左到右，组件越拆越散、依赖越来越外置、能扛的规模越来越大、运维也越来越重。 Lite 适合"今天就想跑通第一个向量检索"，Standalone 适合"一台机器扛住一条业务线"，Cluster 适合"几亿向量、多团队共用、不能停机"。 关键是它们共享同一套 API 与数据模型，所以选型不必一步到位——先用 Lite/Standalone 起步，规模涨上来再迁到 Cluster，业务代码几乎不动。

三者的核心区别其实就是"组件聚合度"和"依赖外置程度"的连续谱：Lite 把一切塞进一个进程、零外部依赖；Standalone 仍是单进程但开始用上对象存储这类依赖； Cluster 则把每个组件拆开、把每个依赖都外置成独立集群，换来无上限的横向扩展。它们共享同一套 API 和数据模型，所以你可以从 Lite 起步、平滑迁移到 Cluster，业务代码几乎不用改。

怎么选？给个简单的决策直觉：在笔记本上学习、跑 demo、写单元测试，用 Lite——一个 pip 装好，数据落在本地文件，几分钟就能跑通第一个向量检索； 数据量在千万级以内、单机内存/磁盘扛得住、不需要高可用，用 Standalone——一个进程搞定，运维成本最低； 数据上亿、并发高、要求高可用、需要按组件独立扩容，上 Cluster——在 K8s 上把 proxy、querynode、datanode 等按各自负载独立伸缩，把 etcd/对象存储/Pulsar 都用生产级集群。 记住这条主线：规模越大，越要把组件拆开、把依赖外置；而无论哪一档，API 和数据模型都不变，这正是 Milvus 让你"小处起步、大处不愁"的底气。

控制面 vs 数据面

前两部分讲清了"向量是什么"和"数据怎么存"。从这一课起，我们把镜头拉到整个集群：Milvus 由一群进程协作完成工作， 而它们之间最根本的分工，就是控制面（control plane）与数据面（data plane）——前者决定谁该做什么， 后者真正把活干掉。理解了这条主线，后面 RootCoord、DataCoord、QueryCoord、Proxy 各司其职的故事才会条理分明。

🔌 生活类比
把 Milvus 想成一座机场：塔台不搬一件行李、不开一架飞机，但它掌握全局——哪架飞机走哪条跑道、停哪个登机口、几点起飞， 这是控制面。而飞机和地勤才是真正运送旅客与货物的人，他们听从塔台调度，但塔台从不替他们扛箱子，这是数据面。 塔台关心的是"调度与元数据"，飞机关心的是"执行与吞吐"。两者一旦混在一起，机场就会瘫痪。

两个面：决策与执行

Milvus 把所有进程按职责切成两层。控制面（协调者 / coordinators）管元数据、调度、分配：集合的 schema 长什么样、 段（segment）该分到哪台机器、哪个 QueryNode 负责哪个分片、什么时候触发 flush 与 compaction……它们做决定，但几乎不碰用户的真实数据流。 数据面（节点 / nodes）则是干活的人：Proxy 接收请求、QueryNode 在内存里做向量检索、DataNode 把写入落盘成段、StreamingNode 承载写入日志。 它们听协调者的安排，但真正的吞吐压力都压在它们身上。换句话说，控制面回答的是"该怎么办"，数据面回答的是"现在就办"； 前者是一张随时更新的"作战地图"，后者是地图上正在冲锋的部队。本课接下来要做的，就是把这张地图的图例（接口）、指挥部的编制（MixCoord）、以及部队与指挥部如何通信（服务发现）一一讲清。

这种"控制面窄、数据面宽"还有一个微妙之处值得点破：Proxy 虽属数据面，却不存任何用户数据。它是"无状态网关"——只做校验、鉴权、限流、分派， 把请求拆给协调者要时间戳、向节点要结果，自己不持有段也不持有索引。正因如此，Proxy 可以像 Web 服务器一样随意加减实例，前面挂个负载均衡即可。 这提醒我们：控制面/数据面是按"是否承载用户吞吐"划分的，而不是按"是否存数据"。Proxy 在数据面，是因为它站在吞吐链路上； 而它能轻松扩缩，又恰恰因为它无状态。把"承载吞吐"和"持有状态"这两件事分开看，你才能准确理解每个组件的扩缩容策略。

这种分层不是为了好看，而是为了可扩展与高可用：决策逻辑（元数据、调度）天然是"少而关键"的，适合集中、强一致地管理； 执行逻辑（检索、写盘）天然是"多而繁重"的，适合横向铺开、按负载扩容。把两者拆开，你就能单独扩展数据面（加 QueryNode 提升查询吞吐） 而不必动控制面，反之亦然。这正是"存算分离"在进程层面的延伸——上一部分讲的 etcd、对象存储、消息队列，是外部依赖； 这一部分讲的控制面/数据面，是Milvus 自身进程的分工。

分层还带来一个常被忽视的好处：故障隔离。数据面节点是"无状态"的——它内存里的索引和段，都是从对象存储里加载、可随时重建的副本， 真正的"权威元数据"只在控制面与 etcd 里。于是一台 QueryNode 突然崩溃，丢的只是"一份内存副本"，控制面立刻把它负责的分片重新分给别人， 对外几乎无感。反过来，控制面虽然关键，但它处理的请求量小、逻辑稳定，可以用更保守的方式（如多副本、强一致存储）保护。 把"易变易崩的重活"和"稳定关键的决策"分开，正是分布式系统抗故障的基本功。

再给一组直觉上的数量级对比：一个集群里，协调者通常只有个位数个实例（甚至就一个 MixCoord），而 QueryNode / DataNode 可以有几十上百个。 请求量上，控制面每秒处理的多是"建表、加载、上报心跳"这类低频操作，而数据面每秒要扛成千上万条 search/insert。 这种"控制面窄、数据面宽"的形状，决定了二者必须用完全不同的设计取向——前者求稳求准，后者求快求多。

控制面Coordinators
RootCoord（DDL 与身份）、DataCoord（写入侧与存储）、QueryCoord（读取侧）。掌管元数据、时间戳、段与分片的分配与调度；做决定，不扛吞吐。今天打包进一个 MixCoord 进程。
数据面 · 网关Proxy
所有客户端请求的唯一入口：校验、鉴权、限流，再把任务分派给协调者与节点。无状态，可横向扩展。
数据面 · 执行QueryNode / DataNode / StreamingNode
QueryNode 在内存里做检索；DataNode 把写入落盘成段、跑 compaction 与建索引的实际工作；StreamingNode 承载写入日志（WAL）。真正的 CPU/内存/IO 压力都在这一层。
外部依赖etcd · 对象存储 · 消息队列
第 8 课讲过：元数据/服务发现给 etcd，数据与索引给对象存储，写入日志给消息队列。控制面与数据面都站在它们之上。

组件接口：types.go 里的契约

这套分工不是口头约定，而是写死在 Go 接口里的。internal/types/types.go 用一组接口定义了"每个组件长什么样"。 最底层是一个共同基座 Component：任何组件，无论协调者还是节点，都必须实现 Init / Start / Stop / Register 四个方法—— 初始化、启动、停止、向 etcd 注册自己。这四个动词就是一个分布式组件的"生命周期最小集"。

在 Component 之上，每类组件再叠加自己的业务方法：RootCoord 管 DDL，DataCoord 管段， QueryCoord 管加载，Proxy 是网关，QueryNode / DataNode 是执行节点。 其中最特别的是 MixCoord：它同时嵌入了三个协调者的 gRPC 服务接口，等于"一个接口、三种角色"。

用接口而不是具体类型当"契约"，在工程上有两个实打实的好处。其一是可替换：同一个接口，本地调用时是直连实现，跨进程时是 gRPC 客户端代理， 上层代码完全不必关心对方在本机还是在另一台机器——这正是 internal/distributed/ 那层 gRPC 包装能"无缝插入"的前提。 其二是可测试：接口能被 mock 出来，于是测 Proxy 的逻辑时不必真的拉起一个 MixCoord，给它一个假的实现即可。 你会看到源码里大量 //go:generate mockery 注释，就是在为这些接口自动生成 mock。接口是分布式系统的"可插拔接缝"，这条思路贯穿整个 Milvus。

MixCoord：一个进程，三个角色

很多人第一次看 Milvus 架构图会数出"三个协调者进程"，但今天的真实部署里，它们被打包进同一个进程——MixCoord。 也就是说，RootCoord、DataCoord、QueryCoord 在逻辑上是三个角色，但在物理上是一个可执行体。 这个打包逻辑在 internal/coordinator/mix_coord.go 里：一个 mixCoordImpl 同时持有三套协调者的实现，对外用 NewMixCoordServer 启动。

为什么这么做？因为三个协调者之间交互极其频繁（DataCoord 要问 RootCoord 要时间戳、QueryCoord 要问 DataCoord 段在哪里）， 把它们放进一个进程能省掉大量跨进程 RPC，部署也更简单。但接口仍然各自独立，意味着将来若有需要，依旧能拆回三个进程而不动业务逻辑。 对外，internal/distributed/mixcoord 把它包成一个 gRPC 服务器——注意 internal/distributed/ 下没有单独的 rootcoord/datacoord/querycoord 目录， 正印证了"三合一"的现状。

这其实是 Milvus 架构演进的一个缩影。早期版本里，三个协调者是三个独立进程，各自启动、各自注册、彼此用 gRPC 通信； 这在概念上很干净，但运维上要管三套进程，且协调者之间频繁的内部调用都得走网络，徒增延迟。后来社区把它们合并成 MixCoord： 对内三套实现共享同一进程、直接函数调用；对外仍暴露三套服务接口，向后兼容。"逻辑上分、物理上合"是一种典型的工程折中—— 它保留了清晰的职责边界（接口不变），又收割了同进程协作的性能与运维红利。所以当你看老一点的设计文档说"DataCoord 进程"时，别困惑： 那是角色名，今天它住在 MixCoord 这套房子里。本教程统一遵循"代码为准"：以仓库里真实的 mixCoordImpl 为现状描述。

控制面在想什么

"这个新集合的元数据存哪？""这批新写入该开一个新段还是续用旧段？""这个 QueryNode 挂了，它负责的分片谁来接？" ——全是决策与分配问题。控制面要的是正确、一致、全局视角，不追求高吞吐。

数据面在想什么

"这条 search 请求，我在自己内存里的索引上算 topK""这批 insert，我追加到当前段的列存里""这个段太碎了，我把它和邻居合并" ——全是执行问题。数据面要的是快、并行、可水平扩容，单点挂掉不影响全局元数据。

服务发现与会话：etcd 怎么把它们粘起来

进程拆成两面之后，新问题来了：Proxy 怎么知道 MixCoord 在哪个地址？QueryCoord 怎么知道现在有几台活着的 QueryNode？ 答案是 etcd 上的会话（session）机制。每个组件启动时执行 Register()，在 etcd 里写下一条带租约（lease）的 session 记录—— "我是 querynode-3，地址在这里，我还活着"。租约要靠定期续约维持，一旦进程崩溃、心跳停止，租约到期、记录自动消失。

另一端，协调者watch这些 session 的目录：有节点上线，watch 立刻收到"新增"事件；有节点掉线，收到"删除"事件。 于是控制面始终掌握着当前活着的节点清单，可以据此分配段、做均衡、在故障时把工作转移给别人。这套"注册—续约—监听"就是 Milvus 的服务发现， 也是控制面/数据面能动态协作的底座。它的实现我们留到第 14 课细看，这里先记住它的角色。

这里有个容易混淆的点：session 里存的是"地址"，不是"数据"。etcd 在这条链路上扮演的是"通讯录 + 心跳板"，它告诉你"谁在、在哪"， 但真正的 search/insert 流量绝不经过 etcd——那会瞬间把它打爆。组件拿到对方地址后，是点对点直连 gRPC 发请求的。 把 etcd 想成机场的航班信息屏：它显示哪架飞机在哪个登机口，但旅客是走廊桥直接登机，而不是挤到信息屏前面去。 这条"控制信息走 etcd、数据流量走直连"的分界，和本课的控制面/数据面是同一种哲学在不同层面的复现。

三条边界纪律：别让两个面互相越界

有了大图，最后给三条"边界纪律"，它们能帮你在读后面源码时迅速判断"这段逻辑该属于哪个面"。第一条：权威元数据只属于控制面。 集合 schema、段的归属、分片的分配，这些"真相"只能由协调者写、由 etcd 与元数据存储持久化；数据面节点哪怕在内存里缓存了一份，也只是只读副本， 随时可丢、随时可重建。任何"节点擅自改元数据"都是设计上的越界。

第二条：用户数据吞吐只属于数据面。一条 search 的几千次距离计算、一批 insert 的列存追加，绝不能跑到协调者身上——否则控制面会被海量请求压垮， 全局调度随之停摆。协调者只在"决策时刻"被触达：建表、加载、上报、故障转移，都是低频事件。把高频吞吐挡在控制面之外，是保证集群稳定的红线。

第三条：跨面通信只走接口与 etcd。两个面不靠"共享内存"或"偷看对方文件"协作，而是要么通过 types.go 定义的接口发 RPC， 要么通过 etcd 的 session 与 watch 间接感知。这条纪律换来的是松耦合：你可以独立升级、重启、扩缩任意一个组件，只要它仍遵守接口契约、仍按时续约 session， 整个系统就能继续运转。后面每深入一个协调者，不妨都用这三条纪律回看一眼——它能帮你把纷繁的细节，重新归位到"决策 vs 执行"这条主线上。

🌍 宏观理解
一句话串起来：控制面（协调者）决定谁做什么 → 数据面（节点）真正执行 → types.go 用接口写死契约 → MixCoord 把三个协调者打包成一个进程 → etcd 的 session 让两面互相发现。 这张"决策 vs 执行"的大图，是后面 RootCoord / DataCoord / QueryCoord / Proxy 每一课的共同坐标系。
📌 本课要点
  • 两个面：控制面=协调者（管元数据、调度、分配，做决定）；数据面=节点（Proxy/QueryNode/DataNode，真正干活）。
  • 接口契约：internal/types/types.go 里，Component（Init/Start/Stop/Register）是基座，各组件在其上叠加业务方法。
  • MixCoord：三个协调者今天打包进一个进程（internal/coordinator/mix_coord.go），它嵌入三套 gRPC 服务接口——逻辑三角色、物理一进程。
  • 分层为了扩展：少而关键的决策集中管理，多而繁重的执行横向扩容，两面可独立伸缩。
  • 服务发现：组件用 Register() 在 etcd 写带租约的 session，协调者 watch 它们，实现动态上线/下线感知（详见第 14 课）。

Proxy 网关

上一课我们站在高空俯瞰整个集群：协调者决策、节点执行。这一课聚焦数据面最前线的Proxy（代理 / 网关）—— 所有客户端请求的唯一入口。它既是门卫（校验、鉴权、限流），又是调度员（把请求排进不同队列、分派给协调者与节点）， 还是汇总者（把多个分片的结果合并成一份答案）。读懂 Proxy，你就握住了"一次请求从进门到出门"的全貌。

🔌 生活类比
把 Proxy 想成一家大医院的分诊台：病人（请求）一进门，分诊台先核对身份与挂号（鉴权）、看你今天号还够不够（限流）， 再按病情把你分到不同队列——挂号改信息走"行政窗口"（DDL），打针输液走"治疗区"（DML），做检查看报告走"检验科"（DQL）。 分诊台自己不看病、不拍片，但全院的秩序都靠它维持；它也不保存病历，所以可以随时多开几个窗口分担人流——这正是 Proxy"无状态、可横向扩展"的精髓。

唯一入口：milvuspb 这套 API 表面

无论你用 Python SDK、Go SDK 还是 REST，请求最终都会打到 Proxy 暴露的 gRPC 服务上。这套服务的契约由 proto 定义， 对应包是 milvuspb（来自 milvus-proto/go-api/v3/milvuspb，而非仓库里手写的类型）。一个 CreateCollection、 一个 Insert、一个 Search，在 Proxy 这边都对应一个方法 func (node *Proxy) Xxx(...)。 换句话说，Proxy 就是 MilvusService 这套对外 API 的实现体，而协调者与节点的接口都是"对内"的。

这层"对外/对内"的分界很重要：客户端永远只和 Proxy 说话，看不见也不需要知道后面有几台 QueryNode、段存在哪、TSO 从哪来。 Proxy 把这些复杂度全部封装在身后。于是后端怎么扩、怎么改、怎么做故障转移，对客户端都是透明的——API 稳定，拓扑随意。 这也是为什么我们能在第 8 课说"Lite / Standalone / Cluster 同一套 API"：变的是后端规模，不变的是 Proxy 暴露的这张表面。

再往深一层看，Proxy 还承担了一个容易被忽略的角色：协议与拓扑的翻译官。客户端发来的是"我要在集合 A 里搜这个向量、要 topK=10"这种业务语义， 但后端真正要做的，是"集合 A 现在加载在哪几台 QueryNode 上、它有几个分片、每个分片的 delegator 是谁、要把请求拆成几路发出去"。 这中间的鸿沟，全靠 Proxy 用它缓存的元数据来填平。它要随时跟着集合的加载状态、分片分布而更新这张"路由表"，否则就会把请求发给一台早已不负责该分片的节点。 正因为这份"翻译"工作只读元数据、不改元数据，多个 Proxy 才能各自持有一份缓存、彼此独立地服务，而不需要互相同步——这是它能无状态扩展的前提条件。

三道关卡：校验、鉴权、限流

请求进了 Proxy，并不会立刻被执行，而要先过三道拦截器（interceptor）。这是 gRPC 的标准玩法：在真正的业务方法之前， 串接一组中间件，逐个把关。Milvus 把它们拆成独立文件，职责清晰：

  • 鉴权（authentication）：你是谁？authentication_interceptor.go 校验用户名/密码或 token，认不出来直接拒。
  • 权限（privilege）：你能不能做这件事？privilege_interceptor.go 检查 RBAC 角色对这个集合/操作有没有授权。
  • 数据库路由（database）：你说的是哪个 database？database_interceptor.go 把请求归到正确的 db 上下文。
  • 限流（rate limit）：你今天是不是太猛了？rate_limit_interceptor.go 按配额给写入/查询降速甚至挡回，保护后端不被打垮。

这四关合起来，就是 Proxy 作为"门卫"的全部职责。它们挡在业务逻辑之前，意味着一个非法、越权或超额的请求， 根本到不了排队和执行阶段就被拦下——既省后端算力，也守住了安全与稳定的底线。注意这些都是无状态的判断（查的是元数据与配额）， 所以多开几个 Proxy 实例，每个都能独立完成把关，不需要彼此同步。

其中限流这一关，值得多说一句，因为它体现了 Proxy"保护后端"的自觉。向量数据库的写入与查询都很"重"：一次大批量 insert 可能瞬间灌入几十万行， 一次高 topK、大 nprobe 的 search 可能让 QueryNode 满负荷。如果 Proxy 来者不拒，突发流量会直接把后端打到内存溢出或长尾飙升。 于是限流器按配置的配额（可以按集合、按操作类型、按租户细分）给请求背压（backpressure）：能扛就放行，扛不住就降速、排队甚至直接拒绝并提示稍后重试。 这是一种"宁可让少数请求慢一点、也不让整个集群崩掉"的取舍——把它放在最前端，正是因为越早拦住过载，代价越小。

任务调度器：三个队列各管一摊

过了关卡，请求并不会一窝蜂直接执行，而是被包装成任务（task）放进任务调度器（task scheduler）。 这是 Proxy 的"调度大脑"，它内部维护着三个独立队列，按请求类型分流：

这种"按类型分队列"的设计，背后是一个朴素却深刻的观察：不同请求对系统的诉求根本不同，混在一起只会互相拖累。 建一张表（DDL）一年也许只发生几次，但必须万无一失、严格有序；插入数据（DML）每秒成千上万，要的是高吞吐和持久化保证； 查询（DQL）延迟敏感、彼此独立，要的是极致并发。把它们塞进同一个队列，等于让一辆救护车、一队货车和一群轿车挤同一条单车道——谁都快不了。 分车道，才能各行其速。

  • ddQueue（DDL 队列）：数据定义类——建/删集合、分区、索引、别名。这类操作改的是元数据结构，要严格串行、按序执行。
  • dmQueue（DML 队列）：数据操作类——insert、delete、upsert。这类是写入，要拿时间戳、要写进 WAL/消息队列。
  • dqQueue（DQL 队列）：数据查询类——search、query。这类是读取，要扇出到多个分片再合并结果。

为什么要分三个队列而不是一个大队列？因为这三类请求的排序约束、并发度、资源画像完全不同。DDL 改结构，必须谨慎串行， 否则两个并发的"改 schema"会打架；DML 是写，关心顺序与持久化；DQL 是读，关心并发与延迟，彼此之间几乎无依赖、可以大量并行。 把它们物理隔离，调度器就能对每条流水线施加最合适的策略——读的高并发不会被某个慢 DDL 堵住，反之亦然。 （源码里其实还有第四个 dcQueue"数据控制队列"，用于 flush 这类控制写入状态的操作，这里先聚焦主三条。）

队列内部还藏着一个关键设计：任务的两段式生命周期。一个任务进队后，先经历 PreExecute（执行前校验：把集合名翻译成 ID、检查字段是否匹配、补全默认参数）， 再到 Execute（真正干活：发 RPC、等结果），最后 PostExecute（收尾：归并、组装返回）。 调度器之所以要把"未发出（unissued）"和"执行中（active）"两个阶段分开管理，是为了控制并发——比如 DML 写入要保证拿到的时间戳按序生效， 就不能让后拿到时间戳的任务抢先执行。这套机制把"什么时候允许下一个任务往前走"牢牢攥在调度器手里，是 Proxy 能在高并发下仍维持正确时序的根基。 你不必记住每个方法名，但要记住这个直觉：排队不只是排个先后，更是在为"时间戳顺序"和"资源并发"两件事同时把关。

读与写：两条不同的路

把 DML（写）和 DQL（读）的执行路径并排看，能更清楚 Proxy 的"分派"到底在分什么。它们的终点不同、关心的东西也不同：

DML 写入路径

insert/delete 拿到时间戳后，被写进消息队列 / WAL（按 vchannel 分发），由 StreamingNode/DataNode 异步消费、落盘成段。 Proxy 关心的是顺序与持久化——写成功即返回，真正落盘在后台进行。这正是第 7 课"日志即数据"的写侧入口。

DQL 查询路径

search/query 被扇出（fan-out）到负责各个分片的 QueryNode，每个节点在自己内存里的索引上算出局部 topK， 再把结果汇总回 Proxy 归并（reduce）成全局 topK。Proxy 关心的是并发与延迟——慢的那个分片决定整体快慢。

这条"扇出—归并"是分布式检索的核心难点：单个分片只看见局部数据，必须把它们的局部最优正确地合并成全局最优， 还要处理重复、删除标记、分数归一等细节。它足够复杂，值得单开一课——我们留到第 29 课深挖。这里你只需记住：Proxy 是这次归并的汇合点， 是它把"分散在多台机器上的部分答案"重新拼成"一份完整答案"。

顺带厘清一个常见误解：search 的"一致性级别"也是在 Proxy 这一层落地的。客户端可以要求"强一致"（读到此刻为止的所有写入）或"有界过期"（容忍一点点延迟换更低延时）。 Proxy 的做法是：在发出查询前，决定带上一个保证时间戳（guarantee timestamp），告诉 QueryNode"请把数据至少消费到这个时间点再回答我"。 强一致就用当前最新的时间戳，最终一致就用一个稍旧的。你看，又是时间戳在背后统一调度读与写的可见性——这正是第 3 课埋下的伏笔在分布式层面的兑现。 把这一点和前面的扇出/归并合起来，你就理解了 Proxy 在一次查询里同时扮演的三个角色：路由器、时序协调者、结果归并器。

无状态：Proxy 为什么能随便扩

最后回到那句反复出现的话：Proxy 是无状态的。它不持有任何段、任何索引、任何用户数据——它需要的元数据来自协调者与 etcd（且可缓存）， 时间戳来自 TSO，真正的数据在节点和对象存储里。既然每个 Proxy 实例都能独立、对等地完成"把关—调度—归并"，那么应对流量高峰就很简单： 多开几个 Proxy，前面挂一个负载均衡器，请求随便落到哪个实例都一样。某个 Proxy 崩了也不丢数据，因为它本来就没存数据。

这就是把"承载吞吐"和"持有状态"分开设计的回报（第 9 课讲过这条原则）：Proxy 站在吞吐链路最前端，压力最大，但因为无状态， 它反而是整个系统里最容易扩展的一环。把它和"有状态、需要协调分配"的 QueryNode/DataNode 对照，你会更深刻地体会 Milvus 的分层哲学—— 让每个组件只承担一种复杂度，再用接口与队列把它们干净地缝起来。

不过"无状态"并不等于"无脑转发"。Proxy 仍要维护一份会随后端变化而更新的本地缓存（集合 schema、分片分布、各 QueryNode 的负载提示等）， 否则每个请求都去问一次协调者，协调者就成了新瓶颈。这份缓存是"软状态"——可以丢、丢了能从协调者与 etcd 重新拉回来，所以不破坏无状态扩展的本质。 它的更新同样借助第 9 课讲的 watch 机制：当 QueryCoord 改变了某个集合的分片分布，相关变更会传导到 Proxy，让它的路由表保持新鲜。 理解了这一点，你就能回答一个常见面试题："Proxy 既然无状态，为什么还要缓存？"——因为缓存是为性能服务的软状态，而非为正确性兜底的硬状态，两者在分布式系统里有本质区别。

最后把视角拉回这一部分的主线。Proxy 是数据面的网关，但它一刻也离不开控制面：要时间戳找 RootCoord（下一课），要知道集合加载在哪找 QueryCoord（第 13 课）， 要落盘成段背后是 DataCoord（第 12 课）。换句话说，Proxy 是把"控制面的决策"翻译成"对客户端可用的服务"的那层皮。 接下来三课我们逐个走进这三个协调者，你会发现每一个都在为 Proxy 这条流水线的某一环提供支撑——读完它们，再回看这张图，一切都会严丝合缝。

🌍 宏观理解
一句话串起来：Proxy 是唯一入口（milvuspb 服务）→ 先过鉴权/权限/数据库/限流四关 → 包装成任务进 ddQueue/dmQueue/dqQueue 三个队列 → 向协调者要时间戳 → 写入扇进 WAL、查询扇出到节点 → 归并成一份结果返回。 它无状态，所以可以横向扩展、前挂负载均衡。把这条流水线记牢，后面每个协调者都是它身后的一环。

RootCoord

上一课 Proxy 在排队时反复向"协调者"要一个东西——全局时间戳。发这个时间戳的，正是这一课的主角 RootCoord（根协调者）。 它管两件大事：一是所有 DDL（建/删集合、分区、别名、数据库）与其背后的元数据表；二是给全集群发时间与身份—— 单调递增的 TSO 时间戳，和全局唯一的 ID。它是整个 Milvus 的"定序大脑"。

🔌 生活类比
把 RootCoord 想成一个国家的户籍 + 国家授时中心。一方面它是户籍处：谁出生（建集合）、谁注销（删集合）、改名换户（改 schema / 别名）， 都要在它那本总账（元数据表）上登记，登记完才算数。另一方面它是授时台：全国所有钟表都对它报时——它发出的时间只增不减、绝不重复， 于是天南海北发生的事都能排出一个公认的先后。它还兼着发证件号：每个新生事物都领一个全国唯一的编号。登记、授时、发号——一个机构，三件大事。

第一件大事：DDL 与元数据表

当你 create_collection、加一个分区、建一个别名，或者新建一个 database，这些都属于 DDL（数据定义语言）。 它们改变的不是数据本身，而是数据的结构与身份——集合有哪些字段、主键是谁、向量多少维、分了几个分区。这类信息必须有一份权威、持久、唯一的记录， 否则整个集群就会对"集合长什么样"各执一词。RootCoord 持有的这份记录，就叫元数据表（meta table），接口名是 IMetaTable。

为什么 DDL 一定要集中到 RootCoord 一处串行处理？因为结构变更天然怕并发。设想两个客户端同时给同一个集合改 schema、或一个在建分区另一个在删集合—— 若各改各的，元数据就会自相矛盾，后端节点读到的"集合定义"也会随之分裂。把所有 DDL 收口到一个大脑、按序落账，是保证结构一致性最朴素也最可靠的办法。 这也呼应了上一课 Proxy 把 DDL 单独排进 ddQueue 并严格串行的设计——两端配合，才守住了"结构永远只有一个版本"的底线。

你可能会担心：所有 DDL 都串行，会不会太慢？在实践中并不会成为问题，因为 DDL 是低频但高价值的操作——一个集合的 schema 一旦定下，往往几个月都不会改一次， 真正高频的是 insert 与 search 这类数据面操作，而它们走的是完全不同的并发路径（上一课的 dmQueue / dqQueue）。把"罕见但必须正确"的事情做成严格串行，把"频繁且可并行"的事情放开并发， 这种按频率与一致性要求分而治之的设计哲学，贯穿了整个 Milvus。RootCoord 甘当 DDL 的"独木桥"，正是因为这座桥本来就没多少人走，却必须保证每个过桥的人都不会摔下去。

元数据表本身并不把数据存在内存里就算完，它背靠持久化：每一次登记最终都要写进 etcd（下节会展开"存在哪"）。RootCoord 启动时会从持久层把全量元数据回放到内存， 之后内存里的这份既是读缓存（Proxy 频繁来问"集合 A 的 ID 和 schema 是什么"），又随每次 DDL 同步落盘。于是即便 RootCoord 进程重启，集合定义也不会丢—— 这正是"元数据"与"数据"在可靠性上享受同等待遇的体现。

我们用一次最常见的 create_collection 把这条登记链路走一遍，你会看到 RootCoord 的三件大事其实在一个动作里就同时上场了：要发号、要定序、要落账。 这也解释了为什么"建一张表"看起来只是一行 SDK 调用，背后却牵动着整个定序大脑。

第二件大事：TSO，全集群的统一时钟

TSO（Timestamp Oracle，时间戳预言机）是 RootCoord 最具分量的职责。它对外只承诺一件事，却价值千金：发出的每个时间戳都全局唯一、单调递增。 有了它，分布在不同机器上的写入与查询，就能被放到同一条时间线上排出先后——这正是第 3 课讲"时间戳的角色"、第 10 课 Proxy"拿时间戳"的源头。 没有一个统一时钟，分布式系统就无法回答"这两件事谁先谁后"这个最基本的问题。

这个时间戳在概念上是个混合值：高位是物理时间（墙上时钟的毫秒），低位是一个逻辑计数器，两段拼进一个 64 位整数。物理位保证它大致跟着真实时间走， 逻辑位保证同一毫秒内连发多个也各不相同。你不必记住精确的位宽，只要抓住直觉：它既像时间、又保证唯一且有序。正因如此，比较两个时间戳的大小， 就等于在比较两件事在全局时间线上的先后——这是 Milvus 实现一致性读、读写隔离、以及"读到某个时间点为止的数据"的基石。

TSO 还有一个精妙的工程设计：批量分配 + 高水位持久化。RootCoord 并不会每发一个时间戳就写一次 etcd（那样太慢），而是周期性地把一个高水位（high-water mark）预先推进并持久化到 etcd， 之后在这段已被持久化的区间内，发时间戳就是纯内存操作，极快。一个后台 ticker 定时调用 UpdateTSO 推进并落盘水位。这样既快又崩溃安全： 万一 RootCoord 挂掉重启，它从 etcd 读回的水位一定大于等于之前发出去的任何时间戳，于是绝不会发生"重启后发出的时间戳比之前更小"这种致命倒退。

为什么"绝不倒退"这件事如此重要？因为整个 Milvus 的一致性都建立在"时间戳能比大小"这一假设上。查询会带一个读保证时间戳，约定"我要读到这个时间点为止的所有写入"； 写入则各自带着自己的时间戳落进日志。只要时间戳全局单调，节点就能干净利落地判断"哪些写入该被这次查询看见、哪些还太新应被忽略"。一旦时间戳出现倒退或重复，这套判断就会崩塌： 本该可见的数据被漏掉，或本不该出现的数据提前冒头——一致性就此破功。所以 RootCoord 宁可在持久化上多花一点力气，也要把"单调"这条铁律焊死。这也是为什么授时这件事只能有一个源头： 两个时钟各发各的，永远无法保证全局单调。

这里还藏着一个常见的疑问：既然每次发时间戳都要这么讲究，会不会成为整个集群的瓶颈？答案是不会——恰恰因为有了批量分配，绝大多数取时间戳的请求都在内存里瞬间完成， 只有周期性推进水位才触碰 etcd。换句话说，慢的那一步被摊薄到很低的频率，快的那一步承担了几乎全部流量。这是分布式系统里一种典型的"把昂贵操作批量化、把廉价操作高频化"的取舍， 后面讲段分配、ID 分配时你会反复见到同样的套路。理解了 TSO，就等于拿到了读懂 Milvus 一致性模型的钥匙。

t1Proxy 写入 A 入队，向 RootCoord 取时间戳 ts=100
t2另一台 Proxy 写入 B，取得 ts=101（绝不会再发 100）
t3查询 C 取读保证时间戳 ts=102，可读到 ts<=102 的所有写入
t4后台 ticker 推进高水位并持久化到 etcd（崩溃也不回退）

第三件大事：全局唯一 ID

除了发时间，RootCoord 还发号。集合、分区、段（segment）这些对象都需要一个全集群唯一的 64 位 ID 来标识自己。这件事和 TSO 高度同构： 同样要求唯一、同样为了性能而批量分配——一次性从持久层取一段 ID 区间到内存，之后在区间内发号都是内存操作，用完再取下一段。 于是无论哪台机器、哪个时刻申请，拿到的 ID 都不会撞车。Proxy 要建集合时，就会向 RootCoord 的 idAllocator.Alloc(count) 一次性领走一批 ID。

你或许会问：为什么不用 UUID 这种本地就能生成的随机 ID？因为 Milvus 需要的不只是"唯一"，还要"紧凑且可比较"——64 位整数比 128 位字符串省内存、好索引，递增的序号还天然带一点"谁先创建"的语义。 集中发号虽然多了一次和 RootCoord 的往返，但凭借批量分配，这次往返被摊薄到几乎可忽略，换来的却是全集群整洁、统一、可排序的身份体系。这又是一处"用一点协调成本换取全局秩序"的典型权衡。

把"发时间"和"发号"放在同一处，并非偶然：它们都需要一个全局唯一的发号源，都靠"批量取一段、内存内快速分发、高水位持久化防回退"这套相同的机制。 RootCoord 正是这两件事天然的归属——它本就是集群里那个"唯一、权威、串行"的点。理解了这一层，你就明白为什么我们叫它"定序大脑"： 集群中关于时间与身份的最终裁决，都出自这里。

值得点一句的是 TSO 与 ID 在实现上的同源：在源码里它们底层共用同一套"全局分配器"的思路，连持久化用的 KV 句柄都是同一个（你会在下面的代码里看到 tsoKV 同时喂给两个分配器）。 区别只在于：一个分配的是"会跟着墙上时钟走"的时间戳，一个分配的是"只管递增、不关心物理时间"的纯序号。把它们做成同构的两兄弟，既减少了实现负担，也让"唯一性"这件事有了统一的保证来源。 对使用者而言，你只需记住：凡是需要全局唯一标识或全局统一时序的地方，背后都站着 RootCoord 这个发号台。

DDL 如何传播，以及 RootCoord 住在哪

登记完一条 DDL，仅仅写进元数据表还不够——后端的节点也得知道"集合变了"。RootCoord 会把 DDL 通过一条流式（streaming）通道广播出去， 让相关节点按序消费这条结构变更。这条"DDL 走流式日志"的链路本身很有讲究，但它属于后面讲写路径 / 流式系统的范畴，这里只需记住一个直觉： DDL 不是改完就完事，而是像一条带时间戳的消息一样，被有序地播给需要它的人。这样 schema 变更才能在分布式环境下做到"人人最终看到同一个版本、且顺序一致"。

为什么 DDL 也要带时间戳、也要走有序日志？因为结构变更和数据写入在分布式里其实是同一类问题：都需要让"先后"成为全局共识。设想一次"给集合加字段"的 DDL 和一批 insert 几乎同时发生， 节点必须能判断这批数据到底该按旧 schema 还是新 schema 来解释——靠的正是两者各自携带的 TSO 时间戳。于是 DDL 与 DML 在同一条时间线上被统一定序，节点只需老老实实"按时间戳顺序回放"， 就能得到一个前后一致、人人相同的世界视图。这再次印证了 RootCoord 作为"定序大脑"的核心价值：它把结构与数据都纳入同一套时序，整个集群才有了统一的因果。

最后别忘了上一课的结论：今天的 RootCoord 并不是一个独立进程，而是和 DataCoord、QueryCoord 一起跑在 MixCoord 这一个进程里，扮演其中的"根协调"这一逻辑角色（见第 9 课）。 所以当我们说"向 RootCoord 要时间戳"，物理上是在和 MixCoord 进程里的那段逻辑通信。把它单独成课，是因为它的职责边界清晰且关键： 谁来定序、谁来发号、谁来守元数据——这三件事，认准 RootCoord 就对了。下一课我们转向写入侧的另一个大脑 DataCoord（第 12 课），看段是怎么被分配、封存、落盘的。

🌍 宏观理解
RootCoord 是集群里那个"唯一、权威、串行"的点。它干三件事，且都必须由一处来干：登记 DDL（保证集合结构只有一个版本）、发 TSO 时间戳（给全集群一条统一时间线）、发全局唯一 ID（给万物一个不撞车的身份）。 时间戳与 ID 都用"批量取一段 + 高水位持久化"换来既快又防回退；元数据则背靠 etcd 持久化、启动回放。它和另两个协调者合住在 MixCoord 进程里——逻辑上独立，物理上同居。
📌 本课要点
  • DDL + 元数据：建/删集合、分区、别名、库都收口到 RootCoord 串行登记，落在 IMetaTable，背靠 etcd 持久化。
  • TSO 时间戳：全局唯一、单调递增；概念上是"物理时间 + 逻辑计数"的 64 位混合值，给分布式事件一条统一时间线。
  • 批量 + 高水位：TSO/ID 都批量分配、内存内快速发放，高水位持久化到 etcd，崩溃重启绝不回退。
  • 全局 ID：集合/分区/段的唯一 64 位身份，由 idAllocator.Alloc 批量发放。
  • 住在 MixCoord：今天 RootCoord 是 MixCoord 进程里的一个逻辑角色（第 9 课），DDL 经流式通道有序广播给节点。

RootCoord 管"结构与时间"，这一课的 DataCoord（数据协调者）则管"数据落到哪、怎么落、何时清"。它是写入侧的总调度：把涌进来的行分配到一个个段（segment）里， 在合适的时机封存、落盘、合并、回收，还顺手安排索引怎么建。读懂 DataCoord，你就理解了第 7 课"日志即数据、段即载体"在分布式层面的真正运转。

把 DataCoord 想成一座大型仓库的调度主管。货物（数据行）源源不断到货，他先指挥工人把货码进当前正在装的那个货箱（growing 段）； 箱子装满了，就封箱（sealed）、贴上清单、搬进永久库房（落盘成 binlog）。库房里零碎的小箱子多了，他安排并箱（compaction）腾地方； 过期作废的箱子，他派人清运（GC）。他还得决定每条入库流水线（vchannel）由哪个工人（datanode）负责，以及哪些货要额外编目录（建索引）方便日后快速查找。他从不亲手搬货，但整个仓库的秩序全在他脑子里。

段的分配：从一行数据说起

当一批 insert 经 Proxy 写进来，这些行最终要被组织进段（segment）——Milvus 存储与检索的基本单位（第 7 课）。但"该把这批行放进哪个段"并不是节点自己拍脑袋决定的，而是由 DataCoord 统一分配。 它维护着每个集合、每个分片当前正在增长（growing）的那个段，把新来的行分派进去；当这个段写满了（行数或容量达到阈值），或者攒够久了（超时），就把它封存（seal），再开一个新的 growing 段接着收。

为什么要由一个中心来分配段，而不是每个 datanode 各自为政？因为段是要被全集群一致地引用的：QueryCoord 要知道有哪些段、分别加载到哪台 QueryNode；compaction 要知道哪些小段可以合并；GC 要知道哪些段已经没用。 如果段的产生是去中心、随意的，这些下游协调就会失去统一的"账本"。DataCoord 把"段从哪来、现在是什么状态"这件事收口管理，整个写入侧才有了一致、可追溯的视图。它分配段时会给段打上 commonpb.SegmentState_Growing 的初始状态， 后续每一次状态流转也都由它记账。

这里有一个容易混淆的点值得说清：DataCoord 分配段、记录段的状态，但真正把数据写进段、把段刷成文件的，是 datanode / streamingnode 这些干活的节点。这又是第 9 课"协调者决策、节点执行"的一次具体落地—— DataCoord 是那个发号施令、在账本上勾画的大脑，节点才是搬砖落盘的双手。把"决策"与"执行"分开，写入侧才能既有统一秩序、又能水平扩展出多个 worker 并行干活。

段的大小也是一门学问，而这恰恰是 DataCoord 要权衡的：段太小，则数量爆炸，查询要打开成百上千个文件、元数据也臃肿；段太大，则单段加载慢、compaction 一次要搬动的数据量也大。 于是 DataCoord 用"写满阈值 + 超时"两个条件来折中——高写入量的集合靠"写满"快速封段，低写入量的集合靠"超时"避免一个段半天攒不满、迟迟不能落盘被查询到。这个看似不起眼的"何时封段"决策， 直接影响着集合后续的查询性能与存储开销，是 DataCoord 作为写入侧大脑最日常、也最关键的判断之一。

段的一生：四个状态的旅程

一个段从诞生到消亡，会经历一条清晰的状态机。这些状态名不是随便起的，它们是 proto 里定义好的枚举 commonpb.SegmentState_*（注意：它属于 go-api 的 commonpb 包，不在 datacoord 自己的类型里）。 理解这条状态流转，就理解了数据"从内存里的活跃写入，一步步固化成磁盘上不可变文件"的全过程：

Growing正在增长：DataCoord 把新行不断分配进来，段还在内存里活跃接收写入
Sealed已封存：写满或超时，不再接收新行，等待落盘
Flushing落盘中：正在把段写成 binlog 文件持久化到对象存储
Flushed已落盘：成为不可变的持久段，可被加载、索引、合并
Dropped已废弃：被 compaction 合并掉或被删除，等待 GC 清理其文件

这条链路的关键转折是 Flushed：一旦落盘，段就不可变（immutable）了。这是 Milvus 整个存储模型的基石——不可变意味着可以安全地被多个 QueryNode 同时加载、可以放心地建索引、可以被缓存而不必担心被改写。 所有的"修改"（删除、更新）都不去改已落盘的段，而是以新的写入 + delta 日志的形式追加，查询时再把基础数据与 delta 合并出最新视图。这正是第 7 课"日志即数据"的精髓：只追加、不就地改。 DataCoord 的状态机，就是在严格守护这个"一旦固化便不可变"的契约。

除了主干的四态，还有几个旁支状态：Importing（批量导入专用的中间态）、NotExist（占位/不存在），以及上面提到的 Dropped。 你不必背全，但要抓住主线直觉：段的一生，是一条"从可变到不可变、从内存到磁盘、从活跃到回收"的单向旅程，而 DataCoord 就是这条旅程上每一步的签发人。每一次状态切换它都会落账到元数据里， 这样即便节点重启、协调者切换，集群也能从账本里恢复出"每个段现在走到了哪一步"。

把段状态做成一台显式的状态机，还有一个深层好处：它让"数据现在安不安全"变成一个可被精确判断的问题。一个还在 Growing 的段，数据只在内存里，节点一挂就可能丢；一个已经 Flushed 的段，数据已在对象存储里，怎么挂都能恢复。 于是当用户调用 flush 接口、想确保"我刚写的数据已经持久化"，系统只需检查相关段是否都已到达 Flushed 状态即可给出确定的回答。状态机不只是内部记账，它把"持久性保证"这件抽象的事，落成了一个个看得见、查得到的具体状态。

落盘、合并、回收：三件后台大事

段被封存只是开始，DataCoord 还要操心三件持续运转的后台工作。第一件是 Flush（落盘）：把 sealed 段真正写成 binlog 文件、持久化到对象存储。落盘之后，数据才算"安全着陆"—— 即便所有内存数据丢失，也能从对象存储里重建。Flush 由 DataCoord 调度、datanode 执行，是连接"内存中的活跃写入"与"磁盘上的持久档案"的桥梁。

值得一提的是，一次 flush 并非写出单个文件，而是一组 binlog——列数据、统计信息、删除日志各自分开写。按列拆分让后续查询只读需要的字段，把删除单独存放则维持了"只追加"的纪律。于是一个段在对象存储里其实是一小簇文件，DataCoord 会把它们的路径全部记进元数据，供日后加载、compaction 与 GC 精确引用。

后台工作触发时机由谁执行效果
Flush 落盘段被 seal 后datanode段固化为不可变 binlog，数据安全着陆
Compaction 合并小段堆积 / delta 过多datanode小段并大段、应用 delete，查询更轻快
GC 回收文件无引用且过保留期datacoord从对象存储删除垃圾，回收空间省成本

第二件是 Compaction（合并）。随着不断 flush，集合里会积累大量小段，还会因为删除而产生大量 delta 日志。小段多了，查询要打开的文件就多、开销就大；delta 多了，每次查询都要做更多"基础数据 + 删除"的合并。 于是 DataCoord 周期性地调度 compaction，把多个小段合并成大段、把 delete 日志应用进数据（即所谓 L0 compaction），还有为提升检索局部性的 clustering compaction。合并的具体算法（深入是第 19 课）这里不展开， 但要记住它的价值：compaction 是用后台的整理，换前台查询的轻快。它是 LSM 式"只追加"存储模型必然要付出的代价，也是它能长期保持高性能的秘诀。

compaction 的调度也并非一刀切，而是有多种策略各司其职：有的盯着"小段太多"触发合并，有的专门处理 L0 层里堆积的 delete 日志，有的为提升向量数据的局部性做 clustering 重排，还有的在用户主动要求时做强制合并。 DataCoord 会综合段的数量、大小、delete 比例等信号，挑选合适的时机与策略下发任务——既不能太频繁（白白消耗算力、和写入抢资源），也不能太懒（让小段和 delete 越积越拖慢查询）。这种"什么时候、用哪种方式整理"的拿捏， 正是写入侧性能调优的关键一环。

第三件是 GC（垃圾回收）。compaction 合并掉的旧段、被删除的集合、导入失败的残留——这些文件如果不清理，对象存储会越积越多、白白烧钱。DataCoord 的 garbage_collector 周期性地巡检， 找出那些"已经没有任何元数据引用、且过了安全保留期"的文件，安全地从对象存储删除。这里的分寸很关键：删早了可能误删还在被某个慢查询引用的数据，删晚了又浪费空间，所以 GC 总是保守而谨慎，宁可多留一会儿也不冒错删的风险。

把这三件事放在一起看，你会发现它们共同回答了一个问题：如何让一个"只追加、不就地改"的存储系统长期保持健康？只追加意味着写得快、并发友好、天然可靠，但代价是文件会不断增多、删除只是逻辑标记而非物理移除。 Flush 负责把数据安全固化，compaction 负责把碎片整理、把逻辑删除真正兑现，GC 负责把整理后留下的垃圾清走。三者像一支后勤队伍，在前台查询毫无察觉的情况下，默默维持着存储的整洁与高效。 这也是为什么我们说 DataCoord 是"写入侧的总管"——它操心的不只是"怎么写进去"，更是"写进去之后这堆数据如何长治久安"。

Compaction 合并

把许多小段并成大段、把 delete 日志应用进数据，减少查询要打开的文件数与合并开销。是"整理"，目标是让查询更快。深入见第 19 课。

GC 回收

把没人再引用且过了保留期的文件从对象存储删除，回收空间、节省成本。是"清运"，目标是不让垃圾无限堆积。原则是宁可保守也不误删。

通道分配与索引调度，以及"没有 indexnode"

除了管段，DataCoord 还负责通道（channel）分配：决定每个集合的每条 vchannel（虚拟数据通道）由哪个 datanode / streamingnode 来负责消费与落盘。 通道是写入流的载体，把它们均匀地分给各个节点，写入吞吐才能水平扩展；某个节点挂了，DataCoord 还要把它负责的通道重新分配给别人，保证写入不中断。这套"谁负责哪条通道"的账，同样由 DataCoord 统一记。

通道分配看似只是"派活"，其实暗含着写入侧的负载均衡与故障容错两层考量。负载均衡上，DataCoord 要避免把太多繁忙的通道压在同一个节点，否则它会成为写入瓶颈； 容错上，一旦它通过 etcd 的会话机制（第 9 课、第 14 课会细讲）发现某个 datanode 失联，就必须立刻把那个节点身上的通道转交给健在的节点，让中断的写入尽快恢复。你会发现，这和下一课 QueryCoord 给查询侧做的"段/通道分配 + 再平衡" 在思路上高度对称——一个管写入侧的均衡，一个管查询侧的均衡，但都遵循"中心记账、动态调度、故障转移"这同一套协调者哲学。

最后是索引构建调度。向量要能被高效检索，得先建好 ANN 索引（第 5、6 课）。DataCoord 负责调度这些索引构建任务：哪个段该建索引、用什么参数、派给谁建。 这里有一个常被误解的事实，必须讲清楚：Milvus 今天没有独立的 indexnode 进程。索引构建任务虽然由 DataCoord 调度，但真正执行构建的是 datanode 里的 worker。 在代码里你能看到证据：DataNodeClient 接口直接内嵌了 workerpb.IndexNodeClient——也就是说，"索引节点"的能力被合并进了 datanode，而不是另起一个进程。

type DataNodeClient interface {
    io.Closer
    datapb.DataNodeClient
    workerpb.IndexNodeClient   // 索引构建 worker 直接并入 datanode
}

这条"没有 indexnode"的事实，和第 9 课"MixCoord 把三个协调者合进一个进程"是同一种演进方向：把曾经独立的角色合并，降低部署与运维的复杂度。早期 Milvus 确有独立的 indexnode， 但随着架构演进，索引构建被收进了 datanode worker。如果你读到的旧文档还在讲"indexnode 是一个独立组件"，请以代码为准——代码里 DataNodeClient 内嵌 IndexNodeClient，就是当下的事实。 DataCoord 因此成了写入侧名副其实的总管：段、通道、落盘、合并、回收、索引调度，写数据这件事的方方面面，都在它的账本上。

为什么索引构建特别适合并入 datanode，而不是单独成节点？因为索引构建本质上是一个批处理型的重计算任务：读取一个已 flushed 的段、在上面跑 ANN 建图算法、把结果写回对象存储。 它和 datanode 已经在做的"读写段文件"高度同源，共享同一批数据访问能力与节点资源。把它并进来，既省去了一类进程的部署与监控，又让索引构建能就近复用 datanode 的 IO 与缓存。这正体现了 Milvus 近年架构演进的一条主线： 能合则合，用更少的进程类型承担更多职责，从而让中小规模部署更轻、运维更省心——这与第 8 课 Standalone / Cluster 的形态选择遥相呼应。

🌍 宏观理解
DataCoord 是写入侧的总调度，核心是一台段的状态机：Growing → Sealed → Flushing → Flushed（再到 Dropped）。它分配段、记录每一次状态流转，但落盘、合并、建索引都交给 datanode worker 执行——决策与执行分离。 落盘把数据安全固化成不可变 binlog；compaction 用后台整理换前台查询轻快；GC 保守地清运无人引用的文件；通道分配让写入水平扩展。关键事实：没有独立 indexnode，索引构建 worker 已并入 datanode（DataNodeClient 内嵌 IndexNodeClient）。
📌 本课要点
  • 段的分配：DataCoord 维护每个分片当前的 growing 段，写满或超时就 seal、再开新段；段状态由它统一记账。
  • 状态机：commonpb.SegmentState_* 走 Growing → Sealed → Flushing → Flushed → Dropped；注意它在 go-api 的 commonpb 包，不在 datapb。
  • Flushed 即不可变：落盘后的段只读，删除/更新以新写入 + delta 形式追加——只追加、不就地改（第 7 课）。
  • 三件后台事：Flush 落盘、Compaction 合并小段与应用 delete（深入第 19 课）、GC 保守回收无引用文件。
  • 通道与索引：分配 vchannel 给节点；调度索引构建。没有独立 indexnode——构建 worker 并入 datanode（DataNodeClient 内嵌 IndexNodeClient）。

DataCoord 管"数据怎么写进去、怎么存好"，这一课的 QueryCoordV2（查询协调者）则管"数据怎么被搜出来"。它是读取侧的总调度：把要查的集合加载进 QueryNode 的内存， 把段和分片分配给各个节点，用均衡器让负载均匀，用副本换来高可用与高吞吐，还时刻盯着"当前分布"与"目标分布"的差距去校正。读懂它，你就握住了向量检索在分布式层面的全貌。

把 QueryCoordV2 想成一座大图书馆的阅览室调度主任。库房里（对象存储）躺着海量的书（段），但读者不能直接进库房翻——主任得先把要用的书搬上阅览室的书架（加载进 QueryNode 内存），读者才能快速取阅。 他要决定哪本书放哪个书架（段分配），让每个书架的书量均匀、别挤在一处（均衡）；热门的书还会多备几本副本分放不同书架，这样一个书架坏了别处还能借到（高可用），多人同时借也不必排队（高吞吐）。 他手里始终攥着两张表：现在每个书架上实际有哪些书，和理想情况下应该有哪些书——他的工作，就是不断让前者向后者靠拢。

加载与释放：搜索的前提

第一个要打破的直觉是：在 Milvus 里，数据"存着"不等于"能搜"。一个集合的段安安静静躺在对象存储里，本身是搜不了的；必须先经过 load（加载）——QueryCoord 把这个集合的段与索引调度进若干 QueryNode 的内存， 建立起可供检索的在线副本，集合才进入"可搜索"状态。反过来，release（释放）则把这些内存占用还回去，集合重新变为"只存不搜"。这就是为什么你在用 Milvus 时，建完集合、灌完数据，还要显式 load_collection 之后才能 search。

为什么要有这道"加载"的门槛，而不是让所有数据随时可搜？因为向量检索是内存密集的：ANN 索引要常驻内存才能快（第 5、6 课），而内存是昂贵且有限的资源。如果把所有集合无脑全部加载，内存很快就会被撑爆。 于是 Milvus 把"是否占用宝贵的查询内存"这个决定权，通过 load/release 交到用户手里——你想搜哪个集合，就把哪个加载进来；暂时不用了，就释放掉腾出空间。QueryCoord 正是这道闸门的看守者，它统一掌管"此刻哪些集合在线、各自加载在哪些节点上"。

这道闸门的存在，也让 Milvus 能优雅地支持"数据量远大于内存"的场景。你可以往一个集合里灌入远超单机内存的数据，安心地存在对象存储里；真正要查时，再按需把需要的集合（甚至按分区、按需局部）加载进来。 换句话说，存储容量由对象存储兜底、查询能力由内存决定，两者解耦。这正是第 7 课"存算分离"在读取侧的直接体现——你不必为了能查一点数据，就被迫把全部数据都常驻昂贵的内存里。

加载远不只是"把文件读进内存"这么简单，它是一连串协调动作的总和：要确定这个集合有哪些段、哪些分片；要为每个段、每条分片通道挑选承载的 QueryNode；要把它们的索引也一并就位； 还要在加载完成后，让 Proxy 能查到"这个集合现在加载在哪、分了几个副本、每个分片的 delegator 是谁"，这样查询才能被正确地扇出。这一整套，正是 QueryCoord 作为读取侧大脑要操心的事。

段与分片的分配：谁来搜哪一块

一个大集合往往有成百上千个段、若干个分片，而 QueryNode 有很多台——所以核心问题是：哪个段、哪条分片该交给哪台 QueryNode？QueryCoord 要做两类分配。一类是段分配：把历史的、已落盘的段（第 12 课的 Flushed 段）摊到各个节点的内存里，每个节点负责自己那一摊段的局部检索。 另一类是分片（通道）分配：每个分片对应一条 vchannel，由某台 QueryNode 上的 delegator（分片代理 / shard leader）负责。delegator 是这条分片的"门面"，它既要协调该分片下所有段的查询，又要承接该分片最新的、还没落盘的增量数据（growing 段）。

为什么要区分"段"与"分片/delegator"两层？因为一次查询要看到的，既有历史数据（已落盘的 sealed/flushed 段），也有最新数据（还在内存里流动的 growing 段）。delegator 正是把这两者缝合起来的关键角色： 它代表一条分片，把对历史段的查询扇出给持有这些段的 QueryNode，再把这条分片上最新的增量也一并查上，最后归并成这条分片的完整结果。于是 Proxy 只需把查询发给各分片的 delegator，就能拿到"历史 + 最新"都不漏的答案。这套机制的细节（跨分片归并）会在第 29 课深入，这里先抓住角色分工。

delegator 这个角色还解释了一个常见疑问：为什么 Milvus 能做到"刚写进去的数据立刻就能被搜到"（在合适的一致性级别下）？因为最新的 growing 增量本就由 delegator 直接持有并参与查询，不必等它落盘、不必等它被加载—— 查询打到 delegator 时，它把"已落盘的历史段结果"和"手里最新的内存增量结果"现场合并，于是新旧数据一个都不漏。这正是写入侧（DataCoord 的段流转）与读取侧（QueryCoord 的 delegator）在同一条分片上无缝接力的体现：数据在写侧从 growing 一路走向 flushed，在读侧则由 delegator 始终盯着这条流，保证查询看到的永远是完整的当下。

段分配 segment

把已落盘的历史段摊给各个 QueryNode，每台负责一部分段的局部 topK 检索。关注的是把存量数据均匀铺开，让检索算力水平扩展。

分片分配 delegator

每条分片由一个 delegator 承载，缝合该分片的历史段查询与最新 growing 增量，并对外作为这条分片的查询门面。关注的是让每条数据流都有人负责到最新。

均衡器：让负载不偏不倚

分配不是一次性的。节点会上线下线、集合会增删、各节点的负载会逐渐倾斜——于是需要一个 均衡器（balancer）持续地把段和分片重新摊匀。它的目标朴素而重要：别让某台 QueryNode 又热又挤、另一台却闲着， 因为查询的整体延迟往往被最忙的那台拖累。当有新节点加入，均衡器要把一部分段迁过去让它分担；当某节点离开，它负责的段要被接管到别处；当负载长期不均，也要主动搬运校正。

Milvus 的均衡器有不止一种策略。最简单的是轮询（round-robin）：纯按数量把段均匀分，不管每个段实际多大、多热——实现简单，适合分布相对均匀的场景。更精细的是打分（score-based）策略：综合考虑节点的段数、行数、内存占用等"真实负载"来打分， 把段往负载更轻的节点放。两种策略对应不同取舍：轮询图省事、求均匀的数量，打分图精准、求均匀的真实压力。它们都实现了同一个 Balance 接口，由 QueryCoord 按需选用。

均衡这件事还有一个微妙的代价必须权衡：搬迁本身是有成本的。把一个段从 A 节点挪到 B 节点，意味着 B 要重新从对象存储拉取这个段、重新载入内存、重建必要的状态，期间还要小心别影响正在进行的查询。 所以均衡器不能见到一点点不均就疯狂搬运——那会让集群陷入"反复折腾、净收益为负"的抖动。好的均衡策略懂得设阈值、留余量、慢慢收敛：只有当不均衡程度超过某个门槛、且收益明显大于搬迁成本时才动手。这种"克制的勤奋"，正是分布式均衡设计里最考验功力的地方。

副本：用冗余换高可用与高吞吐

加载一个集合时，你可以让它有多个副本（replica）——也就是把同一个集合的段，在不同的 QueryNode 上各加载一份。副本一举换来两样好东西。其一是高可用（HA）：某个副本所在的节点挂了， 另一个副本还在别处好端端地服务，查询不中断、数据不丢可用性。其二是高吞吐：多个副本可以并行分担查询流量，同样一个集合，两个副本理论上能扛近两倍的并发查询。冗余在这里不是浪费，而是稳与快的来源。

当然副本不是免费的：N 个副本就要 N 倍的查询内存。所以副本数是用户根据"这个集合多重要、查询量多大"来权衡的旋钮——核心业务、超高并发的集合多开几个副本，冷僻的集合一个就够。QueryCoord 的 ReplicaManager 统一管理这些副本：每个副本里有哪些节点、各自承载哪些段，都记在它这本账上。 这呼应了第 9 课"协调者持有真相、节点执行"的分工——副本怎么排布是 QueryCoord 决定并记账的，QueryNode 只管把分到自己头上的段加载好、查好。

这里还藏着一个常被忽略的细节：副本之间并不是简单的"完全镜像"，而是各自独立地承载该集合的全部数据、独立地做均衡。这样设计的好处是，一个副本内部的搬迁、扩缩，不会牵连另一个副本，故障与负载彼此隔离。 当一次查询打进来，Proxy 会从可用的副本里挑一个去执行（通常按负载或就近原则），于是流量天然地被分摊到各副本上。可以说，副本既是"纵向的安全垫"（挂了有备份），也是"横向的扩音器"（并发被放大）——这正是为什么我们把它列为读取侧最重要的可用性与扩展手段。

分布 vs 目标：永不停歇的校正

在深入这套校正机制前，先把这一课出现的几个核心概念列成一张表，理清各自的角色——它们正是 QueryCoordV2 这本"账"里最重要的几栏：

概念是什么角色
distribution 分布当前实际加载现状记录"哪个段此刻真的在哪台节点"
target 目标期望的理想布局声明"应该加载成什么样"，供校正参照
replica 副本集合的一份在线拷贝换取高可用与高吞吐，N 份即 N 倍内存
delegator 分片代理一条分片的查询门面缝合历史段与最新 growing 增量
balancer 均衡器负载再平衡器持续把段/分片摊匀，避免热点

QueryCoordV2 这个"V2"架构里最精髓的设计，是它把状态拆成了两张视图来管理。一张叫分布（distribution）：现在实际是什么样——哪个段此刻真的加载在哪台节点、哪条分片的 delegator 现在在哪。 另一张叫目标（target）：理想中应该是什么样——根据集合的加载请求、副本数、均衡策略算出来的期望布局。两者之间几乎总有差距：节点刚挂、段刚迁、加载刚发起……现实永远在追赶理想。

QueryCoord 的核心循环，就是一群观测者（observer）不断比对这两张表，发现差距就生成动作去抹平它：目标里该有但分布里还没有的段，就调度加载；分布里多出来、目标里已不需要的，就释放；该搬的搬、该补的补。 这种"声明目标 → 持续校正现状向目标收敛"的设计，正是现代分布式系统（如 Kubernetes 的 reconcile）公认的健壮范式：你不去写一长串"先做这步再做那步"的脆弱流程，而是只声明"我想要的终态"，让系统自己一轮一轮地逼近。 目标本身还分当前目标（current target）与下一目标（next target），用来平滑地从旧布局过渡到新布局而不打断正在进行的查询。理解了这套"分布逼近目标"的循环，你就抓住了 QueryCoordV2 区别于老架构的灵魂。

为什么要分"当前"与"下一"两个目标？想象一次大规模的重新加载或均衡：如果直接把目标一步换成全新布局，正在用旧布局执行的查询就可能突然找不到段。两段式目标让系统能先准备好下一目标、待新布局就绪后再平滑切换， 让旧查询走完、新查询走新布局，整个过程对用户无感。这种"双缓冲式"的切换，把"既要变、又不能停"这对矛盾化解得干净利落。至此，从 load 到分配、到均衡、到副本、再到这套分布逼近目标的循环，QueryCoordV2 作为读取侧大脑的全貌就清晰了—— 它和上一课的 DataCoord 一写一读，共同撑起了 Milvus 在分布式层面"既能稳稳存好、又能快快搜出"的两端。

🌍 宏观理解
QueryCoordV2 是读取侧的总调度。它的世界从 load 开始：存着不等于能搜，必须把集合的段与索引加载进 QueryNode 内存才可检索。它做两类分配——把历史段摊给各节点、把每条分片交给一个 delegator（缝合历史段与最新 growing 增量）； 用均衡器（轮询 / 打分两种策略，同一 Balance 接口）持续把负载摊匀；用副本换取高可用与高吞吐。最精髓的是"分布 vs 目标"：观测者不断让"实际加载现状"向"期望布局"收敛——声明终态、持续校正，正是它健壮的根源。
📌 本课要点
  • load / release：存着 ≠ 能搜；必须把集合段与索引加载进 QueryNode 内存才可检索，释放则还回内存。
  • 两类分配：段分配（把历史段摊给各节点）+ 分片分配（每条分片一个 delegator，缝合历史段与最新 growing 增量）。
  • 均衡器：持续把段/分片摊匀，避免热点；有轮询与打分两种策略，同实现 Balance 接口。
  • 副本：同集合在不同节点各加载一份，换来高可用（节点挂了还有副本）与高吞吐（并行分担查询），代价是 N 倍内存。
  • 分布 vs 目标：distribution=实际现状、target=期望布局；observer 持续校正使现状逼近目标（含 current/next target 平滑过渡）。

把元数据系统想成一栋大楼的物业前台 + 档案室。etcd 是前台：每位住户（组件）一搬进来就在登记簿上写下"我是谁、住哪间、还在不在"，并领一张会定期续期的门禁卡（租约）——卡一旦忘了续，前台就知道这人搬走了。 档案室（对象存储）放的是真正笨重的大件——家具、设备（数据与索引文件）；前台只在登记簿里记一句"3 号库房第 7 排放着谁家的钢琴"，绝不会把钢琴搬到前台。 而Catalog 就是档案管理员：楼里所有人要存取档案，都得走他这道标准窗口，而不是自己翻库房。

一张总图：元数据从协调者落到 etcd

先把这一课要讲的东西摆成一摞，你就不会迷路。最上面是协调者——RootCoord/DataCoord/QueryCoord，它们是元数据的生产者与消费者：建表要写 schema、分段要记 segment、加载要存 replica。 它们不直接拼 etcd 的 key，而是调用下一层 metastore 的 Catalog 接口，用 CreateCollection、AddSegment、SaveReplica 这样的"业务动词"读写元数据。 Catalog 再往下，落到 kv 层——把"一个 Collection 对象"序列化成"一串 key/value"，调用 Save/MultiSave/LoadWithPrefix 这样的原语。最底层，kv 才真正把字节写进 etcd（小而关键的元数据）或把大块数据交给对象存储。

这张图的精髓是一句话：协调者说"业务语言"，Catalog 翻译成"键值语言"，kv 把它写进 etcd。每往下一层，抽象就更通用、更不关心业务含义。 正因为有 Catalog 这道翻译，上层 RootCoord 完全不必知道"一个 collection 在 etcd 里被拆成了多少个 key"；而下层换存储后端（比如 etcd 换成 TiKV）时，上层也一行都不用改。这就是分层解耦在元数据系统里的体现。

不妨拿"建一个集合"走一遍全程，感受这条流水线。客户端发来 CreateCollection，Proxy 把它转给 RootCoord；RootCoord 先向自己要一个全局时间戳与若干 ID（第 11 课讲过），把 schema 组装成一个 model.Collection 领域对象，然后调用 RootCoordCatalog.CreateCollection(ctx, coll, ts)。 到了 Catalog，这个对象被拆解：集合本身一条 key、每个字段一条 key、每个分区一条 key……再调用 kv.MultiSave 把这一批 key/value 一次性事务写入 etcd。整条链上，RootCoord 只说了一句"建表"，真正"拆成多少个 key、怎么保证原子性"全被下面两层悄悄消化掉了。这正是好抽象的标志：上层的意图简单，下层的复杂被封装。

etcd 与会话：组件如何互相发现

元数据要"存得住"，组件还要"找得到"彼此。第 9 课提过：每个组件启动时调用 Register()，在 etcd 里写下一条带租约（lease）的 session——"我是 querynode-3，gRPC 地址在这里，我还活着"。 这套实现在 internal/util/sessionutil/session_util.go 里：Session 结构体持有一个 LeaseID，后台 goroutine 不停地 keepalive 续约；一旦进程崩溃、心跳停止，租约到期，etcd 自动删掉这条 session 记录。

另一端，协调者调用 WatchServices 去监听这些 session 的前缀目录：有节点上线，立刻收到"新增"事件；有节点掉线（租约过期），收到"删除"事件。 于是控制面始终握着一份当前活着的节点清单，据此分配段、做均衡、在故障时把工作转交别人。这套"注册 → 续约 → 监听"就是 Milvus 的服务发现——它和元数据共用同一个 etcd，但承载的是"谁在、在哪"的瞬时状态，而非"集合长什么样"的持久元数据。

这里要再强调第 9 课那条红线：session 里存的是"地址"，不是"数据"。etcd 在这条链路上是"通讯录 + 心跳板"，它告诉你"谁在、在哪"，但真正的 search/insert 流量绝不经过 etcd——那会瞬间把它打爆。组件拿到对方地址后，是点对点直连 gRPC 发请求的。控制信息走 etcd、数据流量走直连，这条分界贯穿整个 Milvus。

为什么非要用"租约 + 续约"，而不是简单地写一个 alive=true 的标记？因为分布式系统里最难处理的不是"优雅退出"，而是"突然失联"——进程被 kill -9、机器断电、网线被拔。这些情况下没人有机会去把标记改成 false。 租约的精妙正在于它把"活着"变成一件需要持续付出才能维持的事：只要心跳一停，etcd 自己就会在租约到期时清掉记录，无需任何人善后。这是一种"故障即默认"的设计——系统假设节点随时可能消失，活着才是需要不断证明的例外。理解了这一点，你就理解了几乎所有分布式心跳机制背后的同一套哲学。

还有个实用细节值得一提：所有这些 key（元数据也好、session 也好）都挂在一个统一的 metaRoot 前缀下（由配置决定，如 by-dev/meta）。这样同一套 etcd 集群就能隔离多个 Milvus 实例——各用各的前缀，互不串扰。这也是为什么源码里到处是 LoadWithPrefix、WatchWithPrefix：在 etcd 这种扁平 key/value 空间里，前缀就是命名空间，是 Milvus 组织整张元数据图的基本手法。

metastore 与 Catalog：元数据的"档案接口"

现在看核心抽象。internal/metastore/catalog.go 里定义了几个按协调者切分的接口：RootCoordCatalog 管库/表/分区/别名/凭证/角色； DataCoordCatalog 管段（ListSegments/AddSegment/DropSegment）、channel、索引；QueryCoordCatalog 管加载信息（SaveCollection/SaveReplica/SaveCollectionTargets）。 每个方法都是一个业务动词，参数是 model.Collection、datapb.SegmentInfo、querypb.Replica 这样的领域对象，而不是裸的字节。

留意这份对称：三个 Catalog 接口恰好对应第 11–13 课的三个协调者。RootCoord 管"有什么"（schema/DDL），DataCoord 管"数据在哪"（段/channel），QueryCoord 管"加载了什么"（副本/target）。 每个协调者只通过自己那个 Catalog 持久化属于自己的那片世界，三片几乎不重叠。正是这种干净的分区，让三个角色能被各自独立推理、也能被打包进同一个 MixCoord 进程而元数据互不踩踏。Catalog 的三分，正是你早已学过的职责三分在存储层的镜像。

注意 RootCoordCatalog 的很多方法都带一个 ts typeutil.Timestamp 参数——这是因为 RootCoord 的元数据是多版本的：DDL 操作都挂在一个时间戳上，于是"在某个时间点看到的 schema"可以被精确复现。 历史上这靠一层"快照 KV（snapshot kv）"实现——把同一个 key 的不同版本按时间戳存成多条记录，读时取"不晚于某时刻"的最新版。它让一致性级别、时间旅行式读取成为可能。今天 Catalog 的具体实现（internal/metastore/kv/rootcoord）持有一个 kv.TxnKV，把领域对象拆成 key/value 落进去。

把元数据读写收口到 Catalog 接口，还有一个工程上的大好处：可测试。测 RootCoord 的建表逻辑时，你不必真的拉起一个 etcd——给它一个 mock 的 Catalog，或一个内存版 kv（internal/kv/mem）就行，测试又快又稳。 这和第 9 课"用接口当可插拔接缝"是同一招：凡是会触达外部世界（存储、网络）的地方，都用接口隔一道，于是既能在生产里接真实 etcd，又能在测试里接假实现。你在 Milvus 源码里看到大量 mock_*.go，很多就是为这些 Catalog/kv 接口自动生成的。抽象不只是为了优雅，更是为了能被独立验证。

kv 层：Catalog 脚下的存取原语

Catalog 再往下踩的是 pkg/kv/kv.go 里一组逐层叠加的接口。最底是 BaseKV：Save/Load/Remove/MultiSave/LoadWithPrefix 这些最朴素的增删查。 往上 TxnKV 加了事务（MultiSaveAndRemove——多个写和删要么全成、要么全败，这对"改 schema 同时删旧 key"至关重要）。 再上 MetaKv 加了 CompareVersionAndSwap（按版本做乐观锁）。最上 WatchKV 加了 Watch——这正是服务发现与"派活"机制的底座。

这种"接口套接口"的叠加，和第 9 课的 Component 是同一种工程审美：需要多少能力，就依赖到哪一层。一个只读配置的模块依赖 BaseKV 就够； 要做事务的 Catalog 依赖 TxnKV；要监听变化的服务发现依赖 WatchKV。实现都在 internal/kv/etcd（etcd 版）与 internal/kv/mem（内存版，测试用）。 下面这段就是从 BaseKV 一路叠到 WatchKV 的契约：

这里 MetaKv 上那个 CompareVersionAndSwap（比较版本再交换，即 CAS）值得单独点一下：它是乐观锁的基础。设想两个协调者副本同时想改同一份元数据，如果各写各的，后写的会悄悄覆盖先写的，造成丢更新。 CAS 的做法是"带着我看到的版本号去写"——只有当 etcd 里的版本仍和我读到的一致时，写才成功；若期间已被别人改过（版本变了），写就失败，调用方需重读重试。这把"先读后写"的竞态挡在了存储层，让上层不必自己加分布式锁。事务保证一组写的原子性，CAS 保证一次写的不被覆盖——两者一起，撑起了元数据在并发下的正确性。

什么存哪：etcd 与对象存储的分工

一个常被问到的问题：既然有 etcd 又有对象存储，到底什么进哪个？答案就一条体量与角色的分界——小而关键、要强一致、要被 watch 的"元数据"进 etcd；大而笨重、只需可靠存放的"数据与索引"进对象存储。 元数据里只留一个指向对象存储的路径引用。这样 etcd 永远很"瘦"，watch 与一致性才扛得住；对象存储则随数据量线性膨胀，但它不参与一致性协调，便宜、可无限堆。

这条分界不是洁癖，而是被 etcd 的硬约束逼出来的。etcd 是一个基于 Raft 的强一致存储，每一次写都要在多数派副本间达成共识——这让它可靠，但也"贵"：它对单个 value 的大小、对总数据量都有保守上限，吞吐也远不如对象存储。 如果有人图省事，把一段几百 MB 的向量 binlog 直接塞进 etcd，轻则拖垮 watch 与选主，重则整个集群的元数据层瘫痪。所以"大块数据一律下沉到对象存储、etcd 里只留路径"不是可选项，而是红线。记住这条，你看任何分布式系统的"元数据 vs 数据"分层时，都会立刻抓住要害：把强一致用在最该用的地方，别让它为海量数据买单。

顺带把三个外部依赖在元数据视角下的分工再钉死一次：etcd 管"状态与协调"、对象存储管"大块数据"、消息队列管"写入流水"。第 8 课从部署角度认识过它们，这一课你看到的是它们在元数据链路里的角色。三者各守一摊、互不越界，正是 Milvus 能把"存"和"算"、"控制"和"数据"干净拆开的底层支撑。

watch：协调者派活、节点接活

最后一块拼图，是控制面如何把决策下发给数据面。机制还是 etcd 的 watch，但方向反过来：不是节点向 etcd 注册、协调者监听，而是协调者把"分配方案"写进 etcd，节点 watch 自己那一份。 比如 DataCoord 决定"vchannel-A 由 datanode-2 负责"，就把这条 channel 分配写进 etcd；datanode-2 watch 到属于自己的 key，就开始消费这条 channel。QueryCoord 派段也类似——这正是上一课"distribution 向 target 收敛"在存储层的落地：target 写进 etcd，节点 watch 后照着干。

把整条链收束成一句：etcd 既是"档案柜"（存元数据），又是"公告板"（发分配、做发现）。组件之间几乎不直接喊话定状态，而是"写 etcd / watch etcd"间接协作——这正是第 9 课"跨面通信只走接口与 etcd"那条纪律的物理实现。 读懂了元数据存哪、组件怎么发现彼此、协调者怎么派活，你就拿到了贯穿整个分布式部分的最后一把钥匙。

最后补一个"watch 并不那么简单"的现实。网络会抖动、连接会断，watch 一旦断开重连，中间那几个事件可能就漏掉了。Milvus 的应对是：watch 时记住一个 revision（版本水位），断线重连后从这个水位续看，而不是从头再来； 更稳的做法是"先全量拉一次、再增量 watch"——先用 GetSessions/LoadWithPrefix 取一份当前快照对齐状态，再带着快照的 revision 开 watch，这样既不丢事件、也不会重复处理。你会发现 WatchServices 正是返回了一个 revision 给调用方，就是为此。 这提醒我们：watch 是"通知"，不是"真相"——真相永远是 etcd 里那份当前状态，通知只是帮你少做轮询。带着这个心态去读任何"监听变更"的代码，你就不会被"事件一定不丢"的幻觉误导。

🌍 宏观理解
一句话串起来：协调者说业务语言 → Catalog 翻译成键值语言 → kv 把它写进 etcd（小而关键的元数据）或对象存储（笨重的数据/索引）。 与此并行，组件用 Register() 写带租约的 session、协调者 WatchServices 做服务发现；派活则反向——协调者把分配写进 etcd、节点 watch 接活。 于是 etcd 同时扮演"档案柜 + 公告板"：存得住状态、又能让两个面间接协作。这就是把前五课所有角色粘起来的胶水。
📌 本课要点
  • 三层落地：协调者（业务动词）→ metastore 的 Catalog 接口 → kv 原语 → etcd / 对象存储。每下一层更通用、更不懂业务。
  • Catalog 抽象：internal/metastore/catalog.go 按协调者切成 RootCoordCatalog / DataCoordCatalog / QueryCoordCatalog，屏蔽底层存储；RootCoord 的方法带 ts（多版本/快照）。
  • kv 层：pkg/kv/kv.go 逐层叠加 BaseKV → TxnKV → MetaKv → WatchKV（增删查 → 事务 → CAS/版本 → 监听），实现在 internal/kv/etcd。
  • 什么存哪：小而关键、需强一致与 watch 的元数据进 etcd；笨重的向量/索引进对象存储，元数据里只留引用路径。
  • 服务发现 + 派活：组件 Register() 带租约 session、协调者 WatchServices 监听上下线；派活反向——协调者把分配写进 etcd、节点 watch 接活。
